# Model: Context Injection Architecture

**Created:** 2026-01-17
**Updated:** 2026-03-27
**Status:** Active

---

## What This Is

The system for providing agents (Orchestrators and Workers) with the necessary mental model and state to perform their tasks. It manages the boundary between "Static Guidance" (Skills/Principles) and "Dynamic State" (Backlog/Investigations).

The architecture has a **frontend-agnostic core** (SPAWN_CONTEXT.md generation, Config struct, KB injection, skill embedding in `pkg/spawn/`) wrapped by a **Claude-Code-specific shell** (SessionStart hooks, PreToolUse hooks in `~/.claude/`). This separation enables extension to non-Claude frontends.

---

## How This Works

The system uses a **Hybrid Injection Model** with multiple paths, all using **global retrieval** (query-based search or role-based filtering). The system currently lacks **scoped auto-surfacing** — the ability to tie knowledge to specific resources (files, packages, skills) and auto-inject it when those resources are accessed.

**Frontend portability:** SPAWN_CONTEXT.md content is pure markdown — it works identically whether piped to `claude`, `codex exec`, or sent via OpenCode HTTP API. The spawn pipeline (`pkg/spawn/`) is already frontend-agnostic. Only the interactive session hooks (SessionStart, PreToolUse) are Claude-Code-locked.

### 1. SessionStart Hooks (Interactive Sessions)

Triggered via `SessionStart` hooks in `~/.claude/settings.json`.
- **Purpose:** Provide immediate, high-bandwidth context for human-driven (orchestrator) sessions.
- **Mechanism:** Two bash hooks that delegate to compiled Go:
  - `~/.claude/hooks/load-skill-identity.sh` — loads orchestrator skill from `~/.claude/skills/meta/orchestrator/SKILL.md`
  - `~/.claude/hooks/orient-hook.sh` — calls `orch orient --hook` for dynamic session orientation (throughput metrics, active agents, pending work)
- **Filtering:** Both hooks check `CLAUDE_CONTEXT` and skip for `worker` and `meta-orchestrator` sessions.

### 2. SPAWN_CONTEXT.md (Spawned Agents)

Generated by `pkg/spawn/context.go` and piped to Claude CLI via stdin.
- **Purpose:** Single authoritative context source for all spawned agents (workers and sub-orchestrators).
- **Mechanism:** Go template (`pkg/spawn/worker_template.go`) assembled with:
  - Task description and orientation frame
  - Compiled skill content (via `pkg/spawn/skill_processing.go` from `skillc`-deployed skills)
  - KB context injection (`pkg/spawn/kbcontext.go`) — prior knowledge, constraints, related models
  - Config resolution (backend, model, tier, mode with sources)
  - Rework context, architect designs, beads issue context
- **Delivery:** Multiple paths depending on backend:
  - Claude CLI: `cat CONTEXT.md | claude --dangerously-skip-permissions` (see `pkg/spawn/claude.go:159`)
  - OpenCode: HTTP API via `client.SendMessageInDirectory()` (see `pkg/spawn/backends/headless.go:126`)
  - Codex CLI: `echo CONTEXT | codex exec` (viable, not yet implemented as backend)

### 3. PreToolUse Hooks (Runtime Governance)

Triggered before specific tool calls during agent execution.
- **Purpose:** Runtime enforcement of governance constraints.
- **Mechanism:** Python scripts in `~/.orch/hooks/` that gate or nudge specific behaviors.
- **Examples:** `gate-bd-close.py` (prevent workers from closing issues), `gate-worker-git-add-all.py` (prevent `git add -A`), `nudge-orchestrator-spawn-context.py`.

### 4. Role-Aware Filtering (`CLAUDE_CONTEXT`)

The primary mechanism for distinguishing between levels:
- `CLAUDE_CONTEXT=meta-orchestrator`: Highest level, strategic oversight.
- `CLAUDE_CONTEXT=orchestrator`: Middle level, triage and synthesis focus. Receives SessionStart hooks.
- `CLAUDE_CONTEXT=worker`: Execution level, implementation focus. Receives only SPAWN_CONTEXT.md.

Set by `pkg/spawn/claude.go` for every spawn via `export CLAUDE_CONTEXT=X`.

## Historical Failure Modes (Resolved)

These were active failure modes when the model was created (Jan 2026). All have been resolved by the architecture described above.

### Token Bloat (Role-Blindness) — RESOLVED
- **Was:** Workers received the 86KB Orchestrator skill because hooks ran without checking CLAUDE_CONTEXT.
- **Fix:** Both SessionStart hooks check `CLAUDE_CONTEXT == worker` and exit early.

### Redundant Guidance — RESOLVED
- **Was:** Beads tracking instructions appeared in `bd prime`, the Orchestrator skill, AND SPAWN_CONTEXT.md.
- **Fix:** SPAWN_CONTEXT.md is the single authoritative source for workers. `bd prime` removed from hooks.

### Session Resume Leakage — RESOLVED
- **Was:** Spawned workers were told to "Resume" a previous manual session.
- **Fix:** The old `session-start.sh` that injected handoffs has been replaced by `orient-hook.sh` which generates fresh orientation.

## Constraints

1. **Skip Orchestrator Skill for Workers:** All hooks must verify `CLAUDE_CONTEXT != worker` before injecting static orchestration guidance.
2. **Authoritative Spawn Context:** For spawned agents, `SPAWN_CONTEXT.md` is the source of truth. SessionStart hooks must back off to avoid duplication.
3. **Pressure Over Compensation:** If an agent lacks context, do not manually paste it. Let the failure surface the gap, then update the template or hook.
4. **No Infrastructure Edits in Ops Mode:** Files governing context injection (`settings.json`, hooks) must not be edited while the system is actively being used for orchestration.

## Integration Points

- **With Skills:** `skillc deploy` compiles skills to `~/.claude/skills/`. SPAWN_CONTEXT.md embeds compiled skill content; SessionStart hooks load orchestrator skill directly.
- **With KB:** `pkg/spawn/kbcontext.go` injects prior knowledge, constraints, decisions, and model references into SPAWN_CONTEXT.md.
- **With Beads:** Issue context is injected into SPAWN_CONTEXT.md for agents working on specific issues.
- **With Orient:** `orch orient --hook` provides dynamic state (throughput, active agents, pending work) at session start.

## Retrieval Primitives

The model currently implements two retrieval primitives:

| Primitive | Mechanism | Precision | Best For |
|-----------|-----------|-----------|----------|
| **Global search** | `kb context` query → keyword match → inject into SPAWN_CONTEXT.md | Medium (false positives from generic terms) | Cross-cutting knowledge (decisions, constraints, principles) |
| **Role filtering** | `CLAUDE_CONTEXT` env var → hook skip/include | High (exact role match) | Skill/guidance injection per agent type |

A third primitive exists in other systems (e.g., Context Hub) but is **not implemented** in orch-go:

| Primitive | Mechanism | Precision | Best For |
|-----------|-----------|-----------|----------|
| **Scoped auto-surfacing** | Annotation tied to resource ID → auto-injected at read time | High (exact scope match) | Point-specific learnings about files, packages, skills |

**Gap analysis:** orch-go's gap detection system (`pkg/spawn/gap.go`) identifies missing context and the learning system (`pkg/spawn/learning.go`) tracks recurring gaps, but resolutions flow into the global `kb quick` pool rather than being scoped to the resource where the gap occurred. This creates accumulation pressure (50+ entries needing triage) and precision loss (keyword matching for surfacing).

**See probe:** [Scoped Annotation Loop Pattern](probes/2026-03-18-probe-scoped-annotation-loop-pattern.md) for detailed comparison with Context Hub's approach.

## Frontend Portability

The architecture separates into frontend-agnostic and frontend-locked layers:

| Layer | Frontend-Agnostic? | Notes |
|-------|-------------------|-------|
| SPAWN_CONTEXT.md generation | ✅ Yes | Pure markdown, works with any agent frontend |
| Config struct (`pkg/spawn/config.go`) | ✅ Yes | No Claude-specific fields |
| Backend interface (`pkg/spawn/backends/`) | ✅ Yes | Pluggable — inline, headless, tmux |
| KB context injection (`pkg/spawn/kbcontext.go`) | ✅ Yes | Markdown output, provider-neutral |
| Skill embedding (`pkg/spawn/skill_processing.go`) | ✅ Yes | Compiled skill → markdown |
| SessionStart hooks | ❌ Claude Code only | Load skill + orient. OpenCode equivalent: `experimental.chat.system.transform` plugin |
| PreToolUse hooks | ❌ Claude Code only | Runtime governance. No Codex equivalent exists |
| CLAUDE_CONTEXT env var | ⚠️ Convention | Works with any frontend that inherits env vars |

**Non-Claude frontend paths (tested 2026-03-27):**

- **Codex CLI (v0.116.0):** Loads AGENTS.md fully. Accepts stdin context (`echo ctx | codex exec`). Env vars inherited. No hooks, no governance, no dedicated file tools. Viable for worker spawns, not recommended for orchestrator role.
- **OpenCode TUI + GPT-5.4:** Already 80% wired via existing backends. GPT-5.4 in model snapshot, routes to responses API. Plugin hook system can inject orchestrator skill. `instructions` config can point to arbitrary files. Recommended path for non-Anthropic orchestration.

**See probe:** [Non-Claude Frontend Context Loading](probes/2026-03-27-probe-non-claude-frontend-context-loading.md)

## Open Questions

1. **Context Budget:** What is the total token cost of SPAWN_CONTEXT.md per spawn? Should there be a budget ceiling?
2. **Hook Consolidation:** Should the two SessionStart hooks be merged into a single `orch session-start --hook` command?
3. **Scoped Annotations:** Would a lightweight scoped annotation system (per-file, per-package, per-skill) complement the existing global kb context injection? The infrastructure exists (SPAWN_CONTEXT.md generation, completion pipeline), but the scoping mechanism does not.
4. **GPT-5.4 Protocol Compliance:** The orchestrator skill is ~37k tokens of structured policy. GPT-4o/5.2-codex had 67-87% stall rates. Does 5.4 clear the threshold for orchestrator use? Blocks OpenCode path from production.
