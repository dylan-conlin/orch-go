## Synthesis

**Key Insights:**

1. **Hybrid architecture is optimal** - The current design separates concerns: individual files for editing/tooling, compiled file for reading/context. Splitting distribution would merge these concerns for no benefit.

2. **Context window tradeoffs favor compilation** - Reading 30 files requires 30 tool calls and loses narrative flow. Reading 1 file provides complete context with category structure intact.

3. **Selective inclusion is already solved** - The skillc `includes.principles` feature addresses "I need just these principles" without fragmenting the reference document.

**Answer to Investigation Question:**

The principles.md file should NOT be split at distribution level. The current architecture is already a well-designed hybrid:
- **Source:** Split into `~/.kb/.principlec/src/{category}/{principle}.md` for editing and tooling
- **Distribution:** Compiled into `~/.kb/principles.md` for agent reading and reference

Splitting distribution would:
- Require 30+ file reads vs 1 file read for comprehensive context
- Lose narrative structure (intro, category groupings, provenance table)
- Add complexity to kb context queries
- Provide no benefit since skillc already pulls individual principles

## Implementation Recommendations

### Recommendation Authority

| Recommendation | Authority | Rationale |
|----------------|-----------|-----------|
| Maintain current architecture | architectural | Affects kb, skillc, principlec tools; cross-boundary pattern decision |

### Recommended Approach ⭐

**Maintain Current Hybrid Architecture** - No changes needed. Keep source split, distribution compiled.

**Why this approach:**
- Source files already exist for editing and selective inclusion
- skillc integration already provides per-principle access
- Compiled file provides narrative structure and comprehensive reference
- No functional gap exists that splitting would address

**Trade-offs accepted:**
- 934-line file is large context cost when reading full principles
- Agent cannot easily read "just one principle" without skillc
- Both are acceptable given skillc integration solves selective access

**Implementation sequence:**
1. No implementation needed - current architecture is correct
2. Consider documenting the architecture in a kb quick entry for future reference

### Alternative Approaches Considered

**Option B: Full split (per-principle files as distribution)**
- **Pros:** Granular access, smaller individual files
- **Cons:** 30+ file reads for full context, loses narrative structure, duplicates skillc functionality
- **When to use instead:** Never with current tooling; would be reconsidered if skillc removed

**Option C: Index + individual files (hybrid at distribution level)**
- **Pros:** Provides both overview and individual access
- **Cons:** Adds maintenance burden, index can drift from files, skillc already solves selective access
- **When to use instead:** If kb tooling needed per-principle queries without skillc

**Rationale for recommendation:** The perceived need for split distribution is addressed by existing skillc `includes.principles` feature. Adding split distribution would create redundancy and complexity.
