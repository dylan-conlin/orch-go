# Investigation: orch servers vs orch-dashboard Confusion

**Question:** When should `orch servers` vs `orch-dashboard` be used, and how do we eliminate the confusion in spawn context and documentation?

## Findings

### Finding 1: Two Mechanisms Serve Different Purposes

**Evidence:**

| Aspect | `orch servers` (Go CLI) | `orch-dashboard` (bash script) |
|--------|-------------------------|--------------------------------|
| Scope | ALL projects | orch-go only |
| Tool | tmuxinator | overmind |
| Processes | From tmuxinator YAML | From Procfile |
| For orch-go | 2 procs: api, web | 3 procs: api, web, opencode |
| Purpose | Project dev servers | Orchestrator infrastructure |
| Cleanup | None | Orphan cleanup, stale socket handling |

**Source:** 
- `cmd/orch/servers.go` (uses tmuxinator)
- `~/bin/orch-dashboard` (uses overmind)
- `~/.tmuxinator/workers-orch-go.yml` (2 panes)
- `Procfile` (3 services)

**Significance:** These are NOT competing tools - they serve different purposes. `orch servers` is for generic project dev environments. `orch-dashboard` is specifically for orch-go's orchestrator runtime (which needs OpenCode server).

---

### Finding 2: Spawn Context Gives Wrong Guidance for orch-go

**Evidence:**

`pkg/spawn/context.go` lines around 100:
```go
sb.WriteString("\n**Quick commands:**\n")
sb.WriteString(fmt.Sprintf("- Start servers: `orch servers start %s`\n", projectName))
```

This tells ALL workers (including orch-go workers) to use `orch servers start {project}`.

Meanwhile, `CLAUDE.md` says:
```
**Always use `orch-dashboard` script** - handles orphan cleanup, stale sockets, and proper startup
```

These are contradictory. For orch-go specifically, `orch-dashboard` is the correct command.

**Source:**
- `pkg/spawn/context.go`
- `CLAUDE.md` (Dashboard Server Management section)

**Significance:** Workers spawned for orch-go get incorrect guidance. If they follow spawn context, they'll start servers without OpenCode, and without orphan cleanup. This is the root cause of the confusion.

---

### Finding 3: tmuxinator Config Missing OpenCode

**Evidence:**

`~/.tmuxinator/workers-orch-go.yml`:
```yaml
windows:
  - servers:
      layout: even-horizontal
      panes:
        - orch serve
        - cd web && bun run dev
```

`Procfile`:
```
api: orch serve
web: cd web && bun run dev
opencode: env -u ANTHROPIC_API_KEY ~/.bun/bin/opencode serve --port 4096
```

The tmuxinator config is missing the `opencode serve` process that the Procfile has.

**Source:**
- `~/.tmuxinator/workers-orch-go.yml`
- `Procfile`

**Significance:** Using `orch servers start orch-go` would start servers WITHOUT the OpenCode API server. This would break any tooling that depends on OpenCode being available (like `orch tail`, `orch send`, etc.).

## Synthesis

**Key Insights:**

1. **Different tools for different scopes** - `orch servers` is for ANY project's dev servers (generic). `orch-dashboard` is for orch-go's orchestrator infrastructure specifically. The key distinguisher is: does it need OpenCode? If yes, use `orch-dashboard`.

2. **Spawn context is project-agnostic but orch-go is special** - The spawn context template correctly guides most projects. But orch-go is unique - it's the orchestrator itself and needs OpenCode running. This needs special handling.

3. **The boundary is clear** - Use `orch servers` for project dev servers (frontend, backend, docker). Use `orch-dashboard` for orchestrator infrastructure (api, web, opencode).

**Answer to Investigation Question:**

**When to use each:**
- `orch servers`: For ANY project's development servers (via tmuxinator). Use when you need to run the project's dev environment.
- `orch-dashboard`: ONLY for orch-go. Use when you need the full orchestrator stack including OpenCode.

**To eliminate confusion:**
1. Fix spawn context to detect orch-go and recommend `orch-dashboard` instead of `orch servers`
2. Remove or update `workers-orch-go.yml` tmuxinator config to avoid confusion
3. Document the boundary clearly

## Implementation Recommendations

### Recommendation Authority

| Recommendation | Authority | Rationale |
|----------------|-----------|-----------|
| Fix spawn context for orch-go | implementation | Tactical fix in context.go, no architectural change |
| Remove tmuxinator workers-orch-go.yml | implementation | Cleanup, eliminates confusion |
| Document boundary in guide | implementation | Documentation only |

### Recommended Approach

**Project-Specific Spawn Context** - Detect orch-go in spawn context generation and recommend `orch-dashboard` instead of `orch servers`.

**Why this approach:**
- Minimal change (add condition to context.go)
- Keeps orch servers generic for all other projects
- Workers get correct guidance automatically

**Trade-offs accepted:**
- Special-casing orch-go in code (acceptable - it IS special as the orchestrator)
- tmuxinator config remains (but becomes unused for orch-go)

**Implementation sequence:**
1. Add conditional in `pkg/spawn/context.go` to detect orch-go project
2. Remove or deprecate `~/.tmuxinator/workers-orch-go.yml` 
3. Add guide `.kb/guides/server-management.md` documenting the boundary

### Alternative Approaches Considered

**Option B: Unify into orch servers**
- **Pros:** Single command for everything
- **Cons:** Would need to add overmind support to Go CLI, orphan cleanup logic, OpenCode reuse logic - significant complexity
- **When to use instead:** If we had many orchestrator-like projects

**Option C: Keep as-is, fix docs only**
- **Pros:** No code changes
- **Cons:** Spawn context would still give wrong guidance to workers
- **When to use instead:** If spawn context was read more carefully (it's not - workers follow it literally)

**Rationale for recommendation:** Option A is minimal change that fixes the actual problem (spawn context guiding workers wrong).
