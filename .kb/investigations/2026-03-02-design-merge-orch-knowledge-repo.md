## Synthesis

**Key Insights:**

1. **Global vs project kb must remain distinguishable** — The `kb context --global` workflow relies on `~/.kb` being a cross-project shared resource. Flattening global kb into orch-go's `.kb/` would pollute every project's spawn context with 1,100+ orch-go-specific investigations. The solution is a subdirectory: `.kb/global/` for cross-project knowledge, `.kb/` root for project-specific.

2. **Skills are the highest-value, lowest-risk merge target** — Skill sources (`skills/src/`) have no naming conflicts with anything in orch-go. They're a self-contained directory tree. Moving them and re-running `skillc deploy` from the new path is sufficient. The deployed files are self-healing via `skillc deploy`.

3. **The merge is 90% path updates, 10% structural decisions** — The actual content merge has zero conflicts. The work is updating: (a) `~/.kb` symlink, (b) 6 Go code references, (c) CLAUDE.md references in multiple files, (d) `kb projects.json`, (e) `~/.orch/groups.yaml`, (f) deployed skill source comments (via `skillc deploy`).

**Answer to Investigation Question:**

Merge orch-knowledge into orch-go using a **subdirectory preservation strategy**:

1. **Skills**: Copy `skills/src/` → `orch-go/skills/src/` (new top-level directory in orch-go)
2. **Global KB**: Move global kb contents into `orch-go/.kb/global/` — a new subdirectory that preserves the global/project distinction
3. **Symlink**: Update `~/.kb` → `orch-go/.kb/global/`
4. **Paths**: Update 6 Go code references, CLAUDE.md, projects.json, groups.yaml
5. **Deploy**: Run `skillc deploy` from new location to update all deployed skills
6. **Archive**: Archive orch-knowledge repo (don't delete — preserve git history)

This approach preserves every existing workflow while eliminating the cross-repo friction.

**Options:**
- A: Subdirectory `.kb/global/` within orch-go (symlink `~/.kb` → `.kb/global/`)
- B: Top-level `global-kb/` directory in orch-go (symlink `~/.kb` → `global-kb/`)

- Decision (Share Patterns Not Tools): The `~/.kb` symlink IS the shared contract; where it points is implementation
- Model (spawn-architecture): `kb context --global` filters by `~/.kb` path; it must remain a distinct directory

**RECOMMENDATION:** Option A (`.kb/global/`) because:
- Keeps all knowledge in `.kb/` (single knowledge root)
- Agents already know `.kb/` is the knowledge directory

- If `kb context --global` has issues with nesting, Option B is the fallback (trivial to switch)

## Implementation Recommendations

### Recommendation Authority

| Recommendation | Authority | Rationale |
|----------------|-----------|-----------|
| Merge orch-knowledge into orch-go | strategic | Irreversible repo merge, changes cross-project knowledge topology |
| Global KB as `.kb/global/` subdirectory | architectural | Affects how all projects access global knowledge |
| Skills at `skills/src/` top-level | implementation | Internal orch-go directory structure |
| Simple copy (no git history merge) | strategic | Tradeoff between complexity and provenance |
| Path updates in Go code | implementation | Mechanical code changes |

### Recommended Approach ⭐

**Phased Merge with Subdirectory Preservation** — Merge orch-knowledge into orch-go in 6 ordered phases, keeping global kb distinguishable from project kb.

**Why this approach:**
- Zero-collision content merge (verified empirically)
- Preserves all existing workflows (skillc deploy, kb context --global, spawn)
- Each phase is independently verifiable and rollbackable
- Eliminates daily cross-repo friction for skill edits and knowledge management

**Trade-offs accepted:**
- orch-go repo grows by ~88MB (orch-knowledge size)
- git blame can't trace through to orch-knowledge history (archived repo available)
- Two-step `skillc deploy` workflow continues (cd + deploy)

**Implementation sequence (6 phases):**

#### Phase 1: Merge skill sources (highest value, lowest risk)
1. Create `orch-go/skills/src/` directory
2. Copy all content from `~/orch-knowledge/skills/src/` to `orch-go/skills/src/`
3. Copy `~/orch-knowledge/scripts/` to `orch-go/scripts/skills/` (pre-commit hooks for skill lint)
4. Run `cd orch-go/skills/src && skillc deploy --target ~/.claude/skills` to update deployed skills
5. Verify all 15 skills deploy correctly with new source paths
6. Commit: `feat: merge skill sources from orch-knowledge`

**Rollback:** `git revert` the commit, re-deploy from orch-knowledge

#### Phase 2: Merge global KB
1. Create `orch-go/.kb/global/` directory
2. Copy all content from `~/orch-knowledge/kb/` to `orch-go/.kb/global/`
3. This includes: decisions/, guides/, models/, investigations/, .principlec/, templates/, context/, principles.md, values.md, README.md, projects.json
4. Update `~/.kb` symlink: `ln -sf ~/Documents/personal/orch-go/.kb/global ~/.kb`
5. Verify `kb context --global "test query"` still works
6. Verify `principlec build` works from new location
7. Commit: `feat: merge global kb from orch-knowledge`

**Rollback:** Restore symlink to `~/orch-knowledge/kb`, `git revert`

#### Phase 3: Update Go code paths
1. `pkg/verify/skill_outputs.go:99` — Update `FindSkillManifest()` search path from `~/orch-knowledge/skills/src/worker/` to project-relative `skills/src/worker/` (or detect from cwd)
2. `pkg/spawn/kbcontext.go:25` — Remove `orch-knowledge` from `OrchEcosystemRepos` (content is now in orch-go)
3. `pkg/spawn/ecosystem.go:21` — Same removal
4. `pkg/tmux/follower.go:371` — Remove from ecosystem list
5. Update test files that reference `orch-knowledge`
6. Run `go test ./...` to verify
7. Commit: `refactor: update paths after orch-knowledge merge`

**Rollback:** `git revert`

#### Phase 4: Update configuration and documentation
1. Update `orch-go/CLAUDE.md`:
   - Change `orch-knowledge` reference to note it's now merged
   - Add skill source/distribution split warning from orch-knowledge CLAUDE.md
2. Update `~/.kb/projects.json` — Remove or update orch-knowledge entry
3. Update `~/.orch/groups.yaml` — Remove orch-knowledge from orch group
4. Update `~/.claude/CLAUDE.md` if it references orch-knowledge
5. Update any CLAUDE.md-level references to `~/orch-knowledge` paths
6. Commit: `docs: update references after orch-knowledge merge`

**Rollback:** `git revert`, restore config files from backup

#### Phase 5: Migrate beads issues
1. Audit 10 open orch-knowledge beads issues
2. For relevant issues: create equivalent in orch-go (`bd create ...`)
3. Close migrated issues in orch-knowledge (`bd close ...`)
4. Close stale/irrelevant issues
5. Document migration in commit message

**Rollback:** Issues are append-only, no rollback needed

#### Phase 6: Archive orch-knowledge repo
1. Ensure all content has been merged (diff check)
2. Add README to orch-knowledge: "This repo has been merged into orch-go as of 2026-03-02"
3. Archive the GitHub repo (read-only, preserves history)
4. Do NOT delete — preserves git history and issue history

**Rollback:** Unarchive on GitHub

### Alternative Approaches Considered

**Option B: Monorepo with git subtree**
- **Pros:** Preserves full git history in orch-go
- **Cons:** Complex merge, ongoing subtree push/pull complexity, unclear benefit
- **When to use instead:** If git blame tracing through historical skill edits is critical

**Option C: Keep separate repos, add symlinks**
- **Pros:** Zero migration risk, no structural changes
- **Cons:** Doesn't solve the friction that motivated this merge, cross-repo commits remain
- **When to use instead:** If the merge risk is deemed too high

**Rationale for recommendation:** Option A (phased merge) directly addresses the cross-repo friction while being safely reversible at each phase. The simple copy approach avoids git subtree complexity that adds ongoing maintenance burden for a one-time benefit.
