## Synthesis

**Key Insights:**

1. **skillc architecture is sufficient** - Custom `output` field, dependency resolution, recursive builds all exist. The gap is SKILL.md-specific features (frontmatter handling, different header format).

2. **Migration is incremental** - Skills can be converted one at a time. No big-bang required. Start with one skill, verify orch spawn still works, then batch convert.

3. **No orch-go changes needed** - pkg/skills/loader.go is read-only. As long as skillc produces valid SKILL.md files, orch spawn will work unchanged.

4. **Two migration patterns exist** - Skills with `src/` already use modular sources (convert to `.skillc/`). Skills with single SKILL.md need decomposition (optional - can stay monolithic via single-source skill.yaml).

**Answer to Investigation Question:**

The migration path is:

1. **Extend skillc** to handle SKILL.md-specific patterns (frontmatter preservation or explicit frontmatter.yaml source)
2. **Pilot on one skill** (e.g., investigation) to verify orch spawn compatibility
3. **Convert existing src/ skills** to `.skillc/` convention (feature-impl, codebase-audit)
4. **Optionally convert monolithic skills** or leave them as single-source skill.yaml

## Implementation Recommendations

**Purpose:** Bridge from investigation findings to actionable implementation.

### Recommended Approach: Phased Migration via Epic

**skillc-managed skills** - Migrate skills to `.skillc/` structure incrementally, starting with a pilot.

**Why this approach:**
- Incremental = low risk (revert one skill if broken)
- Leverages existing skillc architecture (minimal new code)
- Maintains backward compatibility (orch spawn unchanged)

**Trade-offs accepted:**
- Two structures coexist during migration (acceptable)
- Some skills may stay monolithic if decomposition isn't valuable

**Implementation sequence:**
1. Enhance skillc manifest for SKILL.md patterns (frontmatter handling)
2. Pilot migration on `investigation` skill
3. Convert modular skills (feature-impl, codebase-audit)
4. Document pattern, optional conversion for remaining skills

### Alternative Approaches Considered

**Option B: All-at-once migration**
- **Pros:** Single conversion effort
- **Cons:** High risk if something breaks; harder to debug
- **When to use instead:** If only 3-4 skills exist

**Option C: Keep manual SKILL.md, skillc only for CLAUDE.md**
- **Pros:** Zero work
- **Cons:** Doesn't solve modular skill problem; duplicated patterns
- **When to use instead:** If skill modularity isn't a real pain point

**Rationale for recommendation:** Finding 3 shows skills already have mixed structures. Phased migration respects this reality and allows validation at each step.
