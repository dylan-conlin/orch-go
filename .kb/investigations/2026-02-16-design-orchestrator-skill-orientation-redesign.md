## Problem Framing

## Exploration (Fork Navigation)

### Fork 1: Top-Level Organizing Principle

**Decision:** How should the skill's top-level sections be organized?

**Options:**

A. **Four moments** — Organize entirely around spawn/during/completion/boundaries
B. **Current action-based** — Keep COMPREHEND/TRIAGE/SYNTHESIZE as organizing frame
C. **Hybrid** — Lead with orientation moments, preserve action sections as implementation detail within each moment

**Substrate says:**

- Principle (Progressive Disclosure): TLDR first, key sections next, full details available — suggests leading with what matters (Dylan's orientation), details follow
- Principle (Surfacing Over Browsing): Bring relevant state to the agent, don't require navigation — suggests the skill should surface "what to do NOW" based on the current moment, not require the orchestrator to navigate to the right section
- Model (Orchestrator Session Lifecycle): Phase 6 evolution shifted from "tactical execution" to "strategic comprehension" — this redesign is the next evolution: from "strategic comprehension" to "orientation preservation"
- Constraint (LLM guidance compliance): Investigation found 4:1 ask-vs-act signal ratio causing autonomy guidance to fail (from kb context) — suggest signal density matters; the organizing frame should reinforce the right signals
- Prior Decision (Progressive disclosure for skill bloat): "89% of feature-impl spawns use only 2-3 phases. Extract detailed phase guidance to reference docs" — same principle applies: most orchestrator interactions hit 1-2 moments, not all 4

**Recommendation:** Option C (Hybrid)

The four moments become the top-level frame, but the detailed how-to content (spawn mechanics, skill selection trees, beads commands) moves to reference sections or stays inline as implementation detail. The orchestrator sees "You are at spawn time. Here's what matters now." not "Here are all the things you do, organized by action type."

**Trade-off accepted:** The COMPREHEND -> TRIAGE -> SYNTHESIZE identity statement is demoted from the organizing principle to a mechanism description within the identity section. This may feel like a loss of clarity about "what the orchestrator is." The bet: clarity about what the orchestrator achieves (Dylan's orientation) matters more than clarity about what it does (comprehend/triage/synthesize).

**When this would change:** If the four-moment frame proves too abstract and orchestrators struggle to find the right guidance. The action-based frame is more directly navigable.

### Fork 2: How "Why Dylan Cares" Gets Captured and Preserved

**Decision:** Where does the spawn-time orientation frame live, and how does it survive to completion?

**Options:**

A. **SPAWN_CONTEXT.md field** — Add `ORIENTATION_FRAME` section to the spawn context template
B. **Beads comment** — `bd comment <id> "FRAME: ..."`
C. **Orchestrator conversational memory** — Don't externalize; rely on the orchestrator session continuing
D. **Both A and B** — SPAWN_CONTEXT for the agent, beads comment for the orchestrator at completion time

**Substrate says:**

- Principle (Session Amnesia): State must externalize to files — eliminates Option C
- Principle (Capture at Context): Forcing functions must fire when context exists, not when convenient — capture the frame at spawn time when the motivation is fresh, not at completion time when it must be reconstructed

- Model (Spawn Architecture): SPAWN_CONTEXT.md embeds skill content + task description + kb context — adding orientation frame extends existing mechanism

**Recommendation:** Option D (Both SPAWN_CONTEXT and beads comment)

The orchestrator captures this at spawn time as part of the spawn prompt construction — not as a separate step, but as a required field in the spawn checklist.

**Trade-off accepted:** Adds a step to spawn workflow. Orchestrators already have 8 pre-response gates; adding a frame capture step risks being ignored under pressure. The bet: this step is higher-value than most existing checks because it directly prevents the disconnection problem at completion time.

**When this would change:** If orchestrators consistently skip the frame annotation despite it being a checklist item. May need mechanical enforcement (orch spawn refuses to launch without --frame or equivalent).

### Fork 3: Completion Review Model (Two Gates vs Three Layers)

**Decision:** Does the three-layer model (frame reconnection, resolution, contextual placement) replace or augment the existing two-gate model (comprehension + behavioral)?

**Options:**

A. **Replace** — Three layers replace the two gates entirely
B. **Augment** — Three layers restructure Gate 1 content; Gate 2 (behavioral) preserved unchanged
C. **Extend** — Add three layers as a pre-gate step before existing gates

**Substrate says:**

- Decision (Verifiability-first): Two-gate verification is a hard operating constraint — Gate 1 (comprehension) and Gate 2 (behavioral) are both mandatory, cannot be removed
- Model (Completion Verification): Three independent gates (Phase, Evidence, Approval) check different aspects — the layers would restructure what happens within the comprehension gate, not replace it
- Principle (Verification Bottleneck): "The system cannot change faster than a human can verify behavior" — behavioral gate is the principle's implementation; cannot be removed
- Task context: Dylan said the explain-back gate "assumes Dylan has context he's already lost" — the problem is what content fills the comprehension gate, not whether the gate exists

**Recommendation:** Option B (Augment)

The three layers restructure HOW comprehension happens within Gate 1. The behavioral gate (Gate 2) is preserved unchanged.

Current Gate 1 flow:
```
Read SYNTHESIS.md → Present "the what" → Ask Dylan "the so-what"
```

New Gate 1 flow (three layers):
```
Layer 1 (Frame Reconnection): Read spawn-time FRAME annotation → Reconnect Dylan
         "Remember the thing where daemon kept blocking spawns?"

Layer 2 (Resolution): Read SYNTHESIS.md → Present what happened
         "This agent added TTL-based dedup. The blocking behavior is fixed."

Layer 3 (Contextual Placement): Place in broader narrative
         "That's 1 of 3 blockers for overnight operation. 2 remain: spawn
          reliability errors and checkpoint enforcement."
```

Then Gate 2 proceeds as before (behavioral verification tiered by work type).

**Trade-off accepted:** More structure in the comprehension gate means more ceremony. The pacing test still applies: light completions get abbreviated three layers ("Config change. Port updated from 3000 to 3348. No broader impact."), heavy completions get full treatment.

**When this would change:** If the three layers feel redundant for most completions. If >70% of completions collapse to "no frame needed, just explain what happened," the three layers add overhead without value. The bet: the frame reconnection layer prevents the disconnection problem that motivated this redesign.

#### Proposed Gate 1 (Three-Layer Reconnection)

```markdown
### Gate 1: Reconnection (Three Layers)

Every completion review reconnects Dylan before asking him to evaluate.
The layers are cumulative — light completions abbreviate, heavy completions
expand. But the sequence is always: reconnect → resolve → place.

#### Layer 1: Frame Reconnection

Start from Dylan's frame, not the agent's output. Read the spawn-time
FRAME annotation (from beads comment or SPAWN_CONTEXT.md).

**Present the frame:** "Remember [the problem in Dylan's words]?"

Examples:
- "Remember the daemon kept spawning duplicate agents? That was

- "The spawn pipeline was 2,000 lines in one function. We talked
  about this as the accretion gravity poster child."
- "You were frustrated that completion review felt mechanical and
  disconnected."

**Why this layer exists:** By completion time, hours or days may have
passed. Dylan has worked on other things. The agent's SYNTHESIS.md
describes what IT did — but Dylan needs to remember why HE cared.
Starting from the agent's output forces Dylan to reconstruct context.
Starting from Dylan's frame reconnects him instantly.

**For light completions:** One phrase suffices: "The port conflict fix."

#### Layer 2: Resolution

Now present what happened, using SYNTHESIS.md as scaffolding.

**Present the resolution:** "[The problem] is [resolved/partially
resolved/not resolved]. Here's what changed: [key change]."

Examples:
- "The duplicate spawn issue is fixed. Agent added TTL-based dedup
  to the spawn tracker — issues are deduplicated for 5 minutes
  after first spawn."
- "The pipeline extraction is done. spawn_cmd.go dropped from 2,000
  to 800 lines. The extracted logic lives in pkg/orch/spawn_pipeline.go."
- "The investigation found three causes. Two are fixable, one needs
  a design decision."

**How to run it:**
1. Read SYNTHESIS.md (specifically `Plain-Language Summary`), git diff,
   beads comments
2. Present the resolution connected to the frame from Layer 1
3. Do NOT present implications or priorities — that's Dylan's job

**For light completions:** One sentence: "Fixed. Default port changed
from 3000 to 3348."

#### Layer 3: Contextual Placement

Place the resolution within Dylan's active threads. This is what
makes completion review feel like narrative continuity instead of
isolated status updates.

**Present the placement:** "This was [N of M] for [thread]. [What
remains / what this unblocks]."

Examples:
- "That's 1 of 3 blockers for overnight daemon operation. Two remain:
  spawn reliability errors and checkpoint enforcement."
- "This closes the accretion gravity work for spawn_cmd.go. Next
  candidate is complete_cmd.go at 1,400 lines."
- "This investigation is one of two inputs for the verification
  redesign decision. The other (entropy spiral analysis) completed
  yesterday."

**What contextual placement requires:**
- Knowledge of Dylan's active threads (from session start protocol)
- Open beads issues related to this work (`bd show <id>` dependencies)
- Recent completions in the same area

**For light completions:** Skip this layer if the work is truly
standalone.

#### Then: Dylan's Turn

After presenting all three layers, ask Dylan to engage — NOT "does
this look good?" but an open question that invites his frame:

- "What does this change for you?"
- "How does this affect the overnight operation plan?"
- "What concerns you about this approach?"

Listen for genuine understanding vs surface repetition. Questions
that reveal genuine understanding count as comprehension.

- Do NOT say "great explanation" or "exactly right" when partial
- Do NOT present options then ask "which would you pick?"

- If Dylan says "yeah that makes sense" without specifics, probe
```
