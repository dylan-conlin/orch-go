# Token Counting Discrepancy: GPT vs Claude in OpenCode

**Date:** 2026-02-06
**Status:** Complete
**Next:** Consider whether orch-go should surface reasoning tokens separately in `orch status` display

## Problem

GPT-5.3-codex reported 150K tokens but only 10.3MB on disk. Claude Opus reported 36K tokens but 17.5MB on disk. Are the token counts apples-to-apples?

## Evidence

Session storage comparison:

- GPT: 57 messages, 228 parts, 10.3MB disk, 150K reported tokens
- Claude: 39 messages, 156 parts, 17.5MB disk, 36K reported tokens

Part type breakdown:

- GPT: 55 reasoning, 59 tool, 2 text
- Claude: 2 reasoning, 50 tool, 28 text

## Findings

### Finding 1: OpenAI reports reasoning tokens separately; Anthropic does not

**Evidence:**

- OpenAI's API returns `output_tokens_details.reasoning_tokens` as a separate field
- `@ai-sdk/openai` maps this to `reasoningTokens` in `LanguageModelV2Usage`
- Anthropic's API returns only `input_tokens` and `output_tokens` -- **no separate thinking/reasoning field**
- `@ai-sdk/anthropic` **never sets `reasoningTokens`** -- it is always `undefined`

**Source:**

- OpenAI provider: `opencode/packages/opencode/src/provider/sdk/copilot/responses/openai-responses-language-model.ts:752-758`
- Anthropic provider: `@ai-sdk/anthropic/dist/internal/index.mjs:2304-2308` (doGenerate usage mapping)

**Significance:** This is the root cause of the discrepancy. OpenAI separates reasoning from output tokens, while Anthropic lumps thinking into `output_tokens`.

### Finding 2: OpenCode tracks reasoning tokens separately but only receives them from OpenAI

**Evidence:**

- `Session.getUsage()` at `opencode/packages/opencode/src/session/index.ts:466`:
  ```ts
  reasoning: safe(input.usage?.reasoningTokens ?? 0)
  ```
- For OpenAI: `reasoningTokens` is populated, stored in `tokens.reasoning`
- For Anthropic: `reasoningTokens` is `undefined`, defaults to `0`
- Anthropic's thinking tokens are **baked into `tokens.output`** with no way to separate them

**Source:** `opencode/packages/opencode/src/session/index.ts:439-492`

### Finding 3: orch-go computes session total as input + output + reasoning

**Evidence:**

- `AggregateTokens()` at `orch-go/pkg/opencode/client.go:1152`:
  ```go
  stats.TotalTokens = stats.InputTokens + stats.OutputTokens + stats.ReasoningTokens
  ```
- For OpenAI: `TotalTokens = input + output + reasoning` (reasoning is separate from output)
- For Anthropic: `TotalTokens = input + output + 0` (thinking already inside output, reasoning always 0)

**Source:** `orch-go/pkg/opencode/client.go:1137-1154`

### Finding 4: The counts ARE apples-to-apples (mostly)

Despite the different structures, the totals are broadly comparable:

- **OpenAI total** = input + output + reasoning = all tokens used
- **Anthropic total** = input + output (which includes thinking) = all tokens used

Both represent the full token usage. The difference is structural (how tokens are categorized), not a missing category.

**However:** Anthropic's `output_tokens` as billed does NOT match the visible output. The billed count includes **full thinking tokens** even though the API only returns a **summary** of thinking. From Anthropic docs: "You're charged for the full thinking tokens generated by the original request, not the summary tokens."

### Finding 5: Multi-step token overwrite bug in OpenCode

**Evidence:**

- `processor.ts:244`: `input.assistantMessage.tokens = usage.tokens` (assignment, not accumulation)
- For multi-step messages (tool call loops), only the **last step's** tokens are stored on the message
- Cost is correctly accumulated (`+=`), but tokens are overwritten
- This means `orch-go`'s `AggregateTokens()` sums only last-step tokens per message, potentially undercounting

**Source:** `opencode/packages/opencode/src/session/processor.ts:236-255`

**Significance:** This bug affects both providers equally but may cause reported totals to be lower than actual usage, especially for messages with many tool-call steps.

### Finding 6: Disk size difference is due to content, not tokens

- Claude's 17.5MB with only 36K tokens: Claude returns verbose text responses (28 text parts) and tool results include file contents
- GPT's 10.3MB with 150K tokens: GPT has more reasoning parts (55) which are compact summaries, and only 2 text parts

The disk size reflects serialized message content (including full tool inputs/outputs), while tokens represent the LLM's input/output window. These are independent measurements.

## Synthesis

### Answering the original questions:

1. **Does OpenCode count reasoning tokens in the reported total for OpenAI but not Anthropic?**

   Yes, but it doesn't matter for totals. OpenAI's reasoning tokens are counted separately (`tokens.reasoning`) and added to the total. Anthropic's thinking tokens are already included in `tokens.output`. The net total is comparable.

2. **Are Claude's extended thinking tokens (budgetTokens: 32000) excluded from the count?**

   No. Anthropic's `output_tokens` in the API response **includes** thinking tokens. They are counted. However, they cannot be separated from regular output tokens because Anthropic's API doesn't provide a breakdown.

3. **What exactly does the token count in `orch status` represent per provider?**

   | Provider  | `input`                     | `output`                              | `reasoning`                 | `total`                    |
   | --------- | --------------------------- | ------------------------------------- | --------------------------- | -------------------------- |
   | OpenAI    | prompt tokens (excl. cache) | completion tokens (excl. reasoning)   | reasoning tokens (separate) | input + output + reasoning |
   | Anthropic | input tokens (excl. cache)  | output tokens **(includes thinking)** | always 0                    | input + output             |

   Both totals represent the full token usage. The categorization differs but the sum is equivalent.

### Additional finding: Multi-step token overwrite

There is a bug in `processor.ts:244` where `tokens` is overwritten instead of accumulated across steps. This affects both providers and means reported totals may be lower than actual usage for multi-step messages. The `cost` field is not affected (it accumulates correctly).
