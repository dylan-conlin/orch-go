## Findings

### Finding 1: Event structure for verification tracking

**Evidence:**
- `verification.failed` events have `gates_failed` array (e.g., ["test_evidence", "git_diff"])
- `agent.completed` events have `verification_passed` (bool), `forced` (bool), `gates_bypassed` (array)
- Both event types include `skill` field

**Source:**
- pkg/events/logger.go:166-197 (VerificationFailedData, LogVerificationFailed)
- pkg/events/logger.go:199-240 (AgentCompletedData, LogAgentCompleted)
- grep of ~/.orch/events.jsonl showing actual event structure

**Significance:** These fields provide all data needed to calculate pass/fail/bypass rates and gate-level breakdown

---

### Finding 2: Real verification metrics reveal gate miscalibration

**Evidence:**
- 174 total verification attempts in last 7 days
- 28.7% pass rate (50 passed first try)
- 27.6% bypass rate (48 used --force)
- test_evidence gate fails 20.1% of attempts (most common)
- feature-impl skill has 22.7% pass rate (lowest)

**Source:** Running `orch stats` with real data from ~/.orch/events.jsonl

**Significance:** High bypass rate suggests gates may be too strict for some workflows (e.g., markdown-only changes triggering test_evidence)

---

### Finding 3: Stats command aggregation pattern

**Evidence:**
- StatsReport struct extended with VerificationStats field
- Gate stats collected in maps during event loop, converted to sorted slices after
- Skill-level stats tracked separately from gate-level stats
- Rates calculated post-loop when TotalAttempts > 0

**Source:** cmd/orch/stats_cmd.go (aggregateStats function)

**Significance:** Follows existing pattern for EscapeHatchStats - collect in maps, convert to sorted slices, calculate rates
