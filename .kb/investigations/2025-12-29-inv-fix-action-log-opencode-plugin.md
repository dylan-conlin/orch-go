## Findings

### Finding 1: Fix is working - new format entries have 0% unknown targets

**Evidence:** 
- Total entries: 509
- Entries with session_id (new format): 14, with unknown target: 0 (0%)
- Entries without session_id (old format): ~495, with unknown target: 436 (~88%)
- Example working entries:
  ```json
  {"timestamp":"2025-12-29T16:09:54.511Z","tool":"Bash","target":"git log --since=\"6am\" --oneline --all 2>/dev/null | head -30","outcome":"success","session_id":"ses_495208f61ffeLwgvNSpPGkePJO"}
  ```

**Source:** `grep -c '"session_id"' ~/.orch/action-log.jsonl` and `grep '"session_id"' ... | grep -c '"target":"unknown"'`

**Significance:** The fix in commit 5f8830fa works correctly. The callID correlation between before/after hooks is successful.

---

### Finding 2: Legacy data from pre-fix sessions pollutes metrics

**Evidence:**
- All 436 entries with `target: "unknown"` lack the `session_id` field
- All 14 entries with proper targets have `session_id` field
- The session_id field was added in the fix (commit 5f8830fa)
- Timeline: unknown entries from ~05:17-16:08, proper entries from 16:09 onward

**Source:** `grep '"target":"unknown"' ~/.orch/action-log.jsonl | grep -v 'session_id' | head`

**Significance:** The 86% unknown rate is a data artifact, not a fix failure. Old sessions loaded old plugin.

---

### Finding 3: OpenCode plugins load at session startup, not dynamically

**Evidence:**
- Plugin changes made during session don't affect current session
- Debug logging added to plugin didn't produce any output until next session
- New session (ses_495208f61ffeLwgvNSpPGkePJO) started logging with new format

**Source:** Observation: modified plugin with debugLog(), no debug log created; but next session used new code

**Significance:** Plugin changes require new OpenCode session to take effect. This explains why fix took time to propagate.

## Synthesis

**Key Insights:**

1. **The fix is verified working** - The OpenCode Plugin API provides `callID` in both `tool.execute.before` and `tool.execute.after` hooks. The fix correctly stores args from the before hook and retrieves them in the after hook using callID as the correlation key. Evidence: 14/14 new-format entries have proper targets.

2. **Legacy data creates misleading metrics** - The 86% unknown rate comes from data generated by sessions that started before the fix was applied. These sessions continued using the old plugin code (which tried to access `output.args` in the after hook where it doesn't exist).

3. **Plugin lifecycle is session-scoped** - OpenCode loads plugins at session startup. This means fixes aren't immediately visible - old sessions must end and new ones start for changes to take effect. This is expected behavior but creates a transition period with mixed data.

**Answer to Investigation Question:**

The fix in commit 5f8830fa IS working correctly. The 86% unknown target rate is due to legacy data from pre-fix sessions, not a bug in the fix. New sessions logging with the fixed plugin have 0% unknown targets. The solution is to either:
1. Archive/rotate the old log file to reset metrics
2. Wait for natural data turnover as new (correct) entries accumulate
3. Add a filter in `orch patterns` to only consider entries with session_id

No code changes needed to the plugin itself.
