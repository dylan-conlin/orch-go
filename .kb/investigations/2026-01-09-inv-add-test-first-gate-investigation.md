## Findings

### Finding 1: Investigation skill has modular source structure

**Evidence:** Investigation skill is built from multiple source files in `~/orch-knowledge/skills/src/worker/investigation/.skillc/`:
- intro.md: Purpose and core principle ("cannot conclude without testing")
- workflow.md: Step-by-step workflow for agents
- template.md: Investigation file template
- self-review.md: Quality checks before completion
- completion.md: Completion protocol

**Source:** `~/orch-knowledge/skills/src/worker/investigation/.skillc/skill.yaml` lines 39-44

**Significance:** To add a test-first gate, I should edit workflow.md since that's where the agent workflow steps are defined. The SKILL.md file is auto-generated by skillc and should not be edited directly.

---

### Finding 2: Current workflow has 6 steps

**Evidence:** The workflow in workflow.md has these steps:
1. Create investigation file
2. IMMEDIATE CHECKPOINT (before ANY exploration)
3. Try things, observe what happens (add findings progressively)
4. Run a test to validate your hypothesis
5. Fill conclusion only if you tested
6. Final commit

**Source:** `~/orch-knowledge/skills/src/worker/investigation/.skillc/workflow.md` lines 1-14

**Significance:** The gate should be inserted between step 2 (IMMEDIATE CHECKPOINT) and step 3 (Try things, observe). This prevents agents from writing elaborate hypotheses before considering a quick test.

---

### Finding 3: Example of investigation theater

**Evidence:** Task description mentions "510-line SvelteKit doc when DevTools check takes 30 seconds" as example of investigation theater.

**Source:** SPAWN_CONTEXT.md line 3

**Significance:** The gate should explicitly ask "Can I test this in 60 seconds?" to prevent agents from diving into documentation when a quick practical test is available.

---

## Synthesis

**Key Insights:**

1. **Modular skill structure enables targeted changes** - The investigation skill's modular design (intro.md, workflow.md, template.md, etc.) makes it easy to add a gate in the right place without affecting other components.

2. **Gate placement is critical** - Inserting the test-first gate between IMMEDIATE CHECKPOINT and "Try things, observe" ensures agents consider quick testing before writing elaborate hypotheses.

3. **60-second rule is concrete and actionable** - By asking "Can I test this in 60 seconds?" the gate provides a clear decision point that prevents investigation theater.

**Answer to Investigation Question:**

The test-first gate was successfully added to workflow.md as step 3, between IMMEDIATE CHECKPOINT (step 2) and "Try things, observe" (step 4). The gate asks agents: "What's the simplest test I can run right now? Can I test this in 60 seconds?" This placement prevents agents from diving into documentation or writing elaborate hypotheses before attempting a quick practical test.
