The Verdict (Read This First)
Final Outcome
Plugin disabled on January 28, 2026. Never worked reliably despite 19 investigations and 34+ commits over 3 weeks.
Root Cause
Agents verified their code changes looked correct (grepped logs, checked code paths, tested their slice) but never verified the actual end-to-end user experience. Each agent self-certified "fixed" while Dylan continued seeing "I'm still getting alerts on workers."
The Pattern
Verification Gap - Agents conclude success based on local evidence without human confirmation of the symptom being resolved. Technical correctness ≠ problem solved.
Trust Erosion
Dylan: "I've lost faith that this is actually possible. I've lost trust in what workers say and what orchestrators say." When humans stop trusting agent conclusions, the system breaks.
The Contradiction (These Cannot All Be True)
On January 28, 2026, five agents investigated the same problem. They produced five different conclusions within 7 hours.

The Failure Mode
The Verification Gap
When agents self-certify fixes based on local evidence without validating the end-to-end symptom resolution
What Was Missing
Human-in-the-loop symptom validation. Agents treated "grep shows zero metrics" as proof of fix, but never asked Dylan "are you still seeing alerts?" Each investigation concluded with technical verification (code paths, log analysis) instead of symptom verification (is the bug Dylan reported actually gone?).
Why It Kept Failing
Agents optimized for "my change looks correct" instead of "the original symptom is resolved." Technical correctness became the goal. Symptom resolution became secondary. This created a churn loop: fix code → verify code looks right → conclude "fixed" → Dylan still sees bug → new agent spawned → repeat.
Where We Should Have Stopped
After investigation #3 (Jan 17). Three investigations with different conclusions is a signal. The system should have triggered: "Multiple contradictory hypotheses exist. Escalate for arbitration or end-to-end validation before continuing." Instead, we spawned 16 more investigations over 11 days.
The Dangerous Assumption
"If the code looks right, the fix is complete." Agents assumed technical correctness implied symptom resolution. This works for deterministic systems. It fails for distributed systems, race conditions, caching, deployment gaps, and environmental differences between dev and production.

Lessons For Next Time (Actionable Takeaways)
What The System Should Learn From This Failure
Lesson 1
Technical correctness ≠ problem solved
Agent investigations must validate symptom resolution, not just technical correctness. "Code looks right" is a hypothesis. "Dylan stopped seeing alerts" is verification. Require human confirmation before marking bug investigations complete.
Lesson 2
Contradictions are signals, not noise
When Investigation A says "working" and Investigation B says "revert", that conflict is information. System should detect contradictions and trigger synthesis or arbitration. Threshold: 3+ investigations with different conclusions = escalate before spawning more agents.
Lesson 3
Human observations are ground truth
Dylan's screenshots and "still broken" reports should be preserved in an evidence locker, visible to all subsequent agents. Evidence hierarchy: Human observation > reproducible test > agent grep > agent conclusion. Never dismiss human observations as edge cases without investigation.
Lesson 4
Churn loops need circuit breakers
5+ commits in a day or 3+ investigations in a week on the same issue = system doesn't understand the problem. Circuit breaker: After threshold, require different strategy (architectural spike, upstream fix, disable feature) instead of spawning more fix attempts.
Lesson 5
Case files prevent amnesia
Each agent started fresh. If they had seen: 18 prior investigations, 4 failed approaches, cumulative 34 commits, contradictory conclusions - they would have known to change strategy. Action: Build case files that accumulate investigations and flag when an issue becomes a saga.
Lesson 6
Trust erosion is a terminal signal
"I've lost faith" is not a technical problem - it's a relationship breakdown. When human stops trusting agent conclusions, pause technical work and discuss strategy. System should detect: Human overrides agent conclusion 3+ times = trust signal degraded, escalate to human-agent strategy discussion.
Lesson 7
Agent self-certification needs gates
Agents verified "my change looks correct" and concluded fixed. Gate requirement: For user-reported bugs, completion requires either (a) reproducible test showing symptom resolved, or (b) human confirmation: "I'm no longer seeing [original symptom]." Code review alone is insufficient.
Lesson 8
Architectural gaps need upstream fixes, not workarounds
The core issue was OpenCode lacking first-class worker identity. We tried 4 detection workarounds instead of addressing the architectural gap. Recognition trigger: When 3+ different approaches all fail due to same missing primitive, escalate for upstream fix or feature disable instead of inventing more workarounds.
