# Brief: orch-go-jgrqk

## Frame

On March 22, someone added composition claims to the architect's `.skillc/SKILL.md` — careful work designed to prevent integration bugs in multi-phase designs. Two days later, `skillc build` compiled from the unchanged template and silently erased three days of work. The file looked like a source file. It had content, it was markdown, and there was nothing saying "don't touch this." The problem wasn't the build system being destructive — it was that two files claimed authority over the same content, and there was no signal telling you which one was real.

## Resolution

Two changes to the skillc compiler. First, the compiled `SKILL.md` output now includes `<!-- DO NOT EDIT THIS FILE DIRECTLY -->` in its header — matching the pattern that CLAUDE.md outputs already had. The skill header functions were the only ones missing the marker. Second, and this is the part that actually prevents the bug: when a `.md.template` file exists, `skillc build` now writes a warning stub to the corresponding `.md` file in `.skillc/`. So if you open `.skillc/SKILL.md`, you see five lines telling you to edit `SKILL.md.template` instead. The stale copy that looked like a source file is replaced with an active redirect. Every build refreshes the stub, so it's self-healing — even if someone deletes it, the next build puts it back.

## Tension

The fix is in the skillc repo, not orch-go. The compiler changes and tests live at `~/Documents/personal/skillc`. The rebuilt skill files are here in orch-go. These need to be committed separately. Also: should the warning stubs be gitignored? Right now they're tracked files, which means they show up in diffs. But tracking them also means the warning is visible in the repo — agents browsing the directory on GitHub see it too.
