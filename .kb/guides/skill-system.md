# Skill System

**Purpose:** Single authoritative reference for how skills work - loading, compilation, and embedding. Read this before debugging skill-related issues.

**Last verified:** Mar 22, 2026

---

## What Skills Are

Skills are instruction sets that tell agents how to do specific types of work:

| Category | Skill | Purpose |
|----------|-------|---------|
| worker | `feature-impl` | Build features with configurable phases |
| worker | `investigation` | Explore codebase, answer questions |
| worker | `systematic-debugging` | Find and fix bugs |
| worker | `architect` | Design decisions and trade-offs |
| worker | `codebase-audit` | Comprehensive review |
| worker | `research` | External research tasks |
| worker | `experiment` | Run experiments |
| worker | `experiential-eval` | Experience-based evaluation |
| worker | `artifact-sync` | Documentation drift sync |
| worker | `design-session` | UI/UX design sessions |
| worker | `ux-audit` | UX quality audit |
| meta | `orchestrator` | Orchestrator session management |
| meta | `exploration-orchestrator` | Parallel decomposition orchestration |
| meta | `exploration-judge` | Judge verdicts on exploration findings |
| meta | `diagnostic` | System diagnostic sessions |
| meta | `meta-orchestrator` | Multi-project orchestration |
| shared | `worker-base` | Common protocols inherited by all workers |

Skills live at: `~/.claude/skills/{category}/{skill}/SKILL.md`

## How Skills Are Loaded

```
orch spawn investigation "task"
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│  pkg/skills/loader.go                                           │
│                                                                 │
│  1. Look for ~/.claude/skills/*/{skill}/SKILL.md               │
│  2. Read entire file content                                    │
│  3. Extract metadata (if any YAML frontmatter)                 │
│  4. Return content for embedding in SPAWN_CONTEXT.md           │
└─────────────────────────────────────────────────────────────────┘
```

**Key insight:** The entire SKILL.md content is embedded in SPAWN_CONTEXT.md. Agents don't load skills dynamically - they receive them at spawn time.

## Skill Compilation (skillc)

Some skills are compiled from multiple source files:

```
orch-go/skills/src/worker/{skill}/.skillc/
├── skill.yaml          # Metadata
├── SKILL.md.template   # Main template
└── phases/             # Phase-specific content
    ├── planning.md
    ├── implementation.md
    └── validation.md
         │
         ▼ skillc deploy
         │
~/.claude/skills/{category}/{skill}/SKILL.md
```

**Auto-generated skills have a header:**
```markdown
<!-- AUTO-GENERATED by skillc -->
<!-- To modify: edit files in .skillc/, then run: skillc deploy -->
```

**Never edit auto-generated SKILL.md directly.** Your changes will be overwritten. Edit the source files instead.

## Template Ownership

| Domain | Owner | Examples |
|--------|-------|----------|
| **Knowledge artifacts** | kb-cli | investigation, decision, guide |
| **Spawn-time artifacts** | orch-go | SYNTHESIS, SPAWN_CONTEXT, FAILURE_REPORT |

Skills reference these templates but don't own them.

## Tiered Spawns

Skills support light vs full tiers:

| Tier | SYNTHESIS.md Required | Use Case |
|------|----------------------|----------|
| **Full** | Yes | Complex work needing synthesis |
| **Light** | No | Quick tasks, investigations |

The `.tier` file in workspace controls this. `orch complete` reads it to know whether to require SYNTHESIS.md.

## Skill Constraints

Skills can declare constraints that get verified at completion:

```markdown
<!-- SKILL-CONSTRAINTS
required:
  - pattern: "*.test.ts"
    description: "Test file must exist"
optional:
  - pattern: "*.md"
    description: "Documentation"
-->
```

`pkg/verify/constraint.go` parses these and checks them during `orch complete`.

## Common Problems

### "Skill not found"

**Cause:** Skill doesn't exist at expected path.

**Check:**
```bash
ls ~/.claude/skills/
ls ~/.claude/skills/worker/  # Most skills are here
```

### "Agent doesn't follow skill guidance"

**Possible causes:**

1. **Skill too long** - Agent may skip/skim if overwhelming
   - Solution: Progressive disclosure (summary first, details in reference)

2. **Conflicting guidance** - Multiple instructions contradict
   - Solution: Review skill for clarity

3. **Guidance buried** - Important instruction not prominent
   - Solution: Move critical guidance to top

### "Edited SKILL.md but changes don't appear"

**Cause:** Skill is auto-generated, your edits were overwritten.

**Check for header:**
```markdown
<!-- AUTO-GENERATED by skillc -->
```

**If present:** Edit source files in `.skillc/`, run `skillc deploy`

**If not present:** Direct edits should work, check spawn is using right skill

### "Orchestrator skill loading in worker sessions"

**Cause:** The session-context plugin auto-loads orchestrator skill for orch projects.

**Fix:** Set `ORCH_WORKER=1` environment variable for worker spawns (orch-go does this automatically).

## Progressive Disclosure Pattern

Skills should follow this structure:

```markdown
# Skill Name

## Quick Reference (TLDR)
- Key points only
- 5-10 lines max

## Core Workflow
- Main process
- When to escalate

## Detailed Phases (collapsed or linked)
- Full guidance for each phase
- Only read when needed

## Reference
- Edge cases
- Examples
```

**Why:** Agents have limited attention. Front-load the important stuff.

---

## Key Decisions (from kn)

- **Skills embedded at spawn time** - not loaded dynamically
- **skillc and orch build skills are complementary** - different purposes
- **Progressive disclosure for bloat** - 77% reduction possible while preserving guidance
- **Skill constraints verified at completion** - SKILL-CONSTRAINTS block
- **ORCH_WORKER=1 skips orchestrator skill** - prevents 37k token waste

## Debugging Checklist

Before spawning an investigation about skill issues:

1. **Check kb:** `kb context "skill"`
2. **Check this doc:** You're reading it
3. **Check skill exists:** `ls ~/.claude/skills/`
4. **Check if auto-generated:** Look for `AUTO-GENERATED` header
5. **Check SPAWN_CONTEXT.md:** Is skill content actually embedded?

If those don't answer your question, then investigate. But update this doc with what you learn.
