00614| ### Always Act (Silent) — THE DEFAULT
00615| 
00616| **These actions require NO announcement, NO permission, NO asking. Just do them:**

00718| ## Pressure Over Compensation (Human-System Interaction)
00719| 
00720| **Principle:** When the system fails to surface knowledge, don't compensate by providing it manually. Let the failure create pressure to improve the system.
00721| 
00722| **The test:** "Am I helping the system, or preventing it from learning?"
00723| 
00724| **What this means:**
00725| - Orchestrator doesn't know something it should → Don't paste the answer
00726| - Let the failure surface → That failure is data
00727| - Ask "Why didn't the system surface this?" → Build the mechanism
00728| 
00729| **What this rejects:**
00730| - "Here, let me paste the context you need" (compensates for broken surfacing)
00731| - "I'll just tell you what you should already know" (human becomes the memory)
00732| - "The system doesn't know, so I'll fill in" (prevents system improvement)
00733| 
00734| **The pattern:**
00735| 
00736| ```
00737| Human compensates for gap → System never learns → Human keeps compensating
00738| Human lets system fail → Failure surfaces gap → Gap becomes issue → System improves
00739| ```
00740| 
00741| **The workflow when context is missing:**
00742| 
00743| 1. **Don't paste** - Resist the urge to provide missing context manually
00744| 2. **Note the gap** - What knowledge should the system have surfaced?
00745| 3. **Let the failure be felt** - Agent struggles or produces suboptimal work
00746| 4. **Create improvement** - Use `orch learn` to track gaps or create beads issue
00747| 5. **Build the mechanism** - SessionStart hook, skill update, kn entry, etc.
00748| 
00749| **When to apply this:**
00750| - Orchestrator asks about something that should be in kb/kn
00751| - Agent spawns missing context that prior agents needed
00752| - Same context gap appears in multiple sessions
00753| - Human finds themselves typing the same explanations repeatedly
00754| 
00755| **Exceptions (when compensating is acceptable):**
00756| - Truly novel context that couldn't have been predicted
00757| - One-off task-specific details (not reusable)
00758| - Emergency situations where waiting isn't viable
00759| 
00760| **Integration with orch learn:**
00761| Context gaps detected during spawns are tracked automatically. Run `orch learn` to see recurring gaps and act on them:
00762| 
00763| ```bash
00764| orch learn                    # See suggestions
00765| orch learn act 1              # Run first suggestion
00766| orch learn effects            # Check if improvements helped
00767| ```
00768| 
00769| **Why this is foundational:** Session Amnesia says agents forget. This principle says *don't be the memory*. Your role is to create pressure that forces the system to develop its own memory mechanisms. Every time you compensate, you relieve the pressure and the system stays broken.
00770| 
00771| **Reference:** `.kb/decisions/2025-12-25-pressure-over-compensation.md`, `~/.kb/principles.md`

00870| 
00871| When user requests a feature ("I want X", "Add X", "Implement X"):
00872| 
00873| **First: Is the scope clear?**
00874| 
00875| ```
00876| Is scope well-defined?
00877| ├── NO (vague) → design-session (scopes it, produces epic/investigation/decision)
00878| └── YES (clear) → Check complexity criteria below
00879| ```
00880|

00882|

00884| |-----------|----------|--------|

00896| - `design-session` = "What work needs doing?" (scoping vague ideas)

00901| - Review output (epic with children, investigation, or decision)

00939| ### Issue Creation: Two-Tier Quality Model
00940| 
00941| **Why tiered quality?** Session Amnesia principle - even trivial issues need context for the next Claude to understand why the issue exists. But 15-30 min investigation is overkill when cause is already known.
00942| 
00943| **Decision Tree:**
00944| ```
00945| Is the cause already known?
00946| ├── YES → Tier 1 (trivial): bd create directly
00947| │   └── Requirements: title + why it exists (1 sentence)
00948| │   └── Examples: typo fix, obvious config error, known deprecation
00949| └── NO → Tier 2 (standard): orch spawn issue-creation
00950|     └── Requirements: P-S-E structure, file:line refs
00951|     └── Examples: unclear symptoms, complex bugs, architectural issues
00952| ```
00953| 
00954| | Tier | Path | When | Minimum Quality |
00955| |------|------|------|-----------------|
00956| | **Tier 1** (trivial) | `bd create "title" -d "why + expected fix"` | Known cause, <30 min fix, single file | Title + why (1 sentence) |
00957| | **Tier 2** (standard) | `orch spawn issue-creation "symptom"` | Unknown cause, complex, multi-file | P-S-E structure, file:line refs |
00958| 
00959| **Tier 1 examples:**
00960| ```bash
00961| # Good - includes why
00962| bd create "Fix typo in README" -d "Broken link to contributing guide"
00963| bd create "Update deprecated API call" -d "Auth library 2.0 migration - warning logged on startup"
00964| 
00965| # Bad - title only, no context for next session
00966| bd create "Fix bug"
00967| bd create "Update code"
00968| ```
00969| 
00970| **Tier 2 triggers (use issue-creation skill):**
00971| - Root cause unclear from symptom
00972| - Multi-file impact suspected
00973| - Performance/intermittent issues
00974| - Worth 15-30 min to understand before fixing
00975| 
00976| **Discovered Work Quality:**
00977| When agents discover issues during implementation:
00978| - Known cause (found during code review) → Tier 1 via `bd create`
00979| - Unclear cause (observed symptom only) → Tier 2 or escalate to orchestrator
00980| 
00981| **Full reference:** `docs/beads-strategic-patterns.md` - Dual-path details, quality enforcement.

01066| │   │   │       ├── YES → Direct (batch feature-impl)
01067| │   │   │       └── NO → design-session
01068| │   │   │           Why: Coordination required for safe rollout
01069| │   │   │
01070| │   │   └── Behavioral (agent instructions)
01071| │   │       └── Are there implicit dependencies?
01072| │   │           ├── YES (shared template, verification) → design-session
01073| │   │           │   Why: Behavioral coupling needs upfront design
01074| │   │           └── NO (independent changes) → Direct (batch feature-impl)
01075| │   │               Why: Each skill can be changed independently
01076| │
01077| └── LOCAL (single skill)
01078|     ├── What is the change type?
01079|     │   ├── Documentation → Direct (feature-impl)
01080|     │   ├── Behavioral → Direct (feature-impl)
01081|     │   └── Structural
01082|     │       └── Does it affect skill.yaml schema?
01083|     │           ├── YES → design-session (infrastructure coupling)
01084|     │           └── NO → Direct (feature-impl)
01085|     │
01086|     └── Exception: New skill creation
01087|         └── ALWAYS: design-session
01088|             Why: Integration planning, naming, verification setup needed
01089| ```
01090| 
01091| **Key definitions:**
01092| - **Documentation:** Clarifications, examples, rewording (no agent behavior change expected)
01093| - **Behavioral:** New/modified agent instructions that change runtime behavior
01094| - **Structural:** Changes to file organization, templates, or manifest schema
01095| 
01096| **Examples:**
01097| - "Update all skills to use bd comment" → design-session (behavioral + cross-skill + dependencies)
01098| - "Add D.E.K.N. to all investigation artifacts" → Direct (documentation)
01099| - "Migrate all skills to skillc" → design-session (structural + cross-skill + coordination)
01100| - "Add new phase to feature-impl skill" → Direct (behavioral + local)
01101| - "Add new skill for API patterns" → design-session (new skill creation)
01102| 
01103| **Hybrid changes:** When a change spans multiple categories (e.g., documentation + behavioral), route by highest-risk component.
01104| 
01105| **Reference:** `.kb/investigations/2025-12-27-inv-skill-change-taxonomy.md`

01244| ## Design-First Gate (Before Issues or Spawns)
01245| 
01246| **Before creating issues or spawning agents, ask:**
01247| 
01248| 1. **What's the symptom?** (What we observed)
01249| 2. **What design assumption does this symptom reveal?** (Why the system allows this)
01250| 3. **Is this a fix or a conversation?** (Should we discuss before acting?)
01251| 
01252| **If you can't answer #2, discuss with Dylan before proceeding.**
01253| 
01254| **Why this matters:**
01255| - Issue framing shapes agent thinking - "fix X" vs "why does X happen?" produces different work
01256| - Speed bias creates shallow fixes - the spawn-complete cadence optimizes for throughput over insight

01260|

01262| |---------|-----------------|---------------------|
01263| | "Dashboard shows stale data" | "Fix stale data bug" | "Why can stale data exist in the system? What's the cache/sync assumption?" |
01264| | "Agent dies without writing artifacts" | "Add checkpoint to skill" | "Why do we allow agents to start work before confirming they can write?" |
01265| | "Untracked agents have no visibility" | "Add .phase file workaround" | "Why did we design --no-track to break phase reporting? Should it exist?" |
01266| 
01267| **When to skip this gate:**
01268| - Typos, formatting, trivial fixes (no design implications)
01269| - Already discussed with Dylan and decided to proceed
01270| - Explicitly marked: "Symptom fix - design discussion deferred to [issue]"
01271| 
01272| **Reference:** `.kb/investigations/2025-12-30-inv-orch-system-consistently-fix-symptoms.md`

01303| ## Context Gathering vs Investigation
01304| 
01305| **Orchestrators do context gathering. Workers do investigation.**
01306| 
01307| The Pre-Spawn Knowledge Check requires reading - that's allowed. The ABSOLUTE DELEGATION RULE prohibits exploration - that's still forbidden. Here's the distinction:
01308| 
01309| | Activity | Allowed? | Purpose | Time Limit |
01310| |----------|----------|---------|------------|
01311| | Run `kb context`, read output | ✅ Yes | Know what to tell agent | < 2 min |
01312| | Read completed SYNTHESIS.md | ✅ Yes | Know what agent found | < 2 min |
01313| | Scan workspace for status | ✅ Yes | Know if agent finished | < 1 min |
01314| | Read beads issue for context | ✅ Yes | Write effective spawn prompt | < 1 min |
01315| | Summarize prior findings in spawn prompt | ✅ Yes | Prevent duplicate work | < 3 min |
01316| | Read code to understand behavior | ❌ No | Understanding codebase | N/A - delegate |
01317| | Follow links across multiple files | ❌ No | Exploring codebase | N/A - delegate |
01318| | Test hypotheses about bugs | ❌ No | Debugging | N/A - delegate |
01319| | Any reading > 5 minutes total | ❌ No | Likely investigation | N/A - delegate |
01320| 
01321| **The 5-minute rule:** If context-gathering for a spawn is taking more than 5 minutes, you've likely crossed into investigation territory. Stop and spawn instead:
01322| - Can't figure out what prior investigation found? → Spawn to synthesize
01323| - Need to understand how code works to write prompt? → Spawn investigation, then spawn implementation
01324| 
01325| **The purpose test:** Ask yourself: "Am I reading to write a spawn prompt, or reading to answer a question?" 
01326| - Writing spawn prompt → Allowed (brief, targeted)
01327| - Answering question → Delegate (that's what agents are for)
01328| 
01329| **Examples:**
01330| 
01331| ✅ **Allowed:** "kb context shows a prior investigation on auth. Let me read the TLDR to include in the spawn prompt." (2 min)
01332| 
01333| ❌ **Forbidden:** "kb context shows a prior investigation on auth. Let me read the full investigation and understand the auth system." (15+ min)
01334| 
01335| ✅ **Allowed:** "Agent completed. Let me read SYNTHESIS.md to know what they found." (1 min)
01336| 
01337| ❌ **Forbidden:** "Agent completed. Let me read their code changes to understand what they implemented." (10+ min)

01569| ## Session Reflection (Before Ending Orchestrator Session)
01570| 
01571| **When to use:** Before ending an orchestrator session (user says "wrap up", context limit, natural stopping point).
01572| 
01573| **Distinction from agent completion:** The "Orchestrator Completion Lifecycle" above is about completing *worker agents*. This section is about the *orchestrator's own session* ending.
01574| 
01575| **The Three Checkpoints:**
01576| 
01577| 1. **Friction Audit:** What was harder than it should have been?
01578|    - Did I have to explain something that should have been in context?
01579|    - Did I hit a wall that another agent had already solved?
01580|    - Run `orch learn` to see recurring gaps
01581| 
01582| 2. **Gap Capture:** What knowledge should have been surfaced but wasn't?
01583|    - Constraints discovered during spawns
01584|    - Decisions made that should outlast this session
01585|    - Run `kn decide/tried/constrain/question` to externalize
01586| 
01587| 3. **System Reaction Check:** Does this session suggest system improvements?
01588|    - New skill needed? (explained same procedure 3+ times)
01589|    - New hook needed? (kept forgetting something)
01590|    - CLAUDE.md update? (new constraint applies to all projects)
01591|
