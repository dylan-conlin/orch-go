  "Homology not isomorphism" ‚ÄĒ what transfers and what doesn't

  Homology means "same shape." Isomorphism means "same thing."

  A bat wing and a human arm have the same bones in the same arrangement
  (homology). But a bat wing can't type and a human arm can't fly (not
  isomorphism).

  What transfers from control theory: The intuitions. Sensor failure cascades.
  The principle that every effective system needs a closed feedback loop. The
  insight that the most catastrophic failure is always removing the sensor,
  because everything else degrades silently.

  What doesn't transfer: The math. Control theory has formal tools for analyzing
   stability (will the system oscillate or settle?), optimality (what's the best
   controller gain?), and observability (can you reconstruct the full system
  state from sensor readings?). These require equations with numbers in them.
  Agent coordination doesn't have those equations. You can't write a transfer
  function for "does this agent understand what correct means."

  - Type systems ‚ÄĒ the compiler makes wrong code unwritable

  observing failures. The placement condition in our experiment is this kind ‚ÄĒ
  someone who understands the codebase decides "function A goes after X,
  function B goes after Y" before agents run.

  Only knowable after failure:
  - Which governance rules agents will bypass (100% bypass rate ‚ÄĒ couldn't

  - Which skill instructions agents will prioritize over others (task alignment

  - Where models of correctness will diverge (stale knowledge cascade ‚ÄĒ agent

  Agents act ‚Üí Failures happen ‚Üí You observe the pattern ‚Üí

  failures accumulate silently until something visible breaks.

  orch-go's daemon is already doing a primitive version of this. It observes
  agent outcomes (completion rate, bypass rate, merge conflicts), detects

  attractor exactly, but the beginning of the loop.

  What's automatable

  Three layers, from easiest to hardest:

  Layer 1: Structural attractor scaffolding (automatable now)

  An orchestrator reads the codebase and decomposes work into non-overlapping
  regions before agents start. This is what orch spawn already does when it
  assigns agents to specific issues targeting specific files. The quality
  depends on whether the decomposition is correct ‚ÄĒ and our experiment proves
  that when it is correct (placement condition), coordination is 100%.

  The gap: the current decomposition is at the issue/file level. The experiment
  shows you need it at the insertion point level for same-file work. Could an

   structural fix? This is harder. The daemon already does a crude version ‚ÄĒ it

  a human-in-the-loop architect decision.

  area. No human needed for the common cases. This is automatable because merge
  conflicts have a specific, parseable structure ‚ÄĒ you can see exactly which

  attractors have gone stale. Which is what the model probe system does ‚ÄĒ

‚ĚĮ i genuinely don't understand why communication-based coordination doesn't

‚Źļ This is the most important question to sit with, because the answer is
  genuinely counterintuitive.

  What your intuition says

  Two smart people are working on the same file. You tell them about each other.
   They discuss their plans. They both agree there's a potential conflict. They
  both say "I'll be careful." They both proceed.

  This should work. With humans, it often does. Your intuition says: awareness +
   discussion + agreement = coordination.

  What actually happens (the raw data)

  In the messaging condition, 18 out of 20 trial pairs did this:

  1. Agent A writes a plan: "I'll insert FormatBytes after FormatDurationShort"
  2. Agent B reads Agent A's plan
  3. Agent B writes a plan: "I'll insert FormatRate after FormatDurationShort"
  4. Both agents read both plans
  5. Both agents note: "The other agent is also inserting after
  FormatDurationShort"
  6. Both agents conclude: "No conflict expected ‚ÄĒ our functions are different"
  7. Both agents insert at the same location
  8. Git merge: CONFLICT

  Step 6 is where it breaks. And it breaks for a reason that's subtle enough to
  be worth unpacking.

  Why "no conflict expected" is true from each agent's perspective

  Agent A is thinking: "I'm adding FormatBytes. Agent B is adding FormatRate.
  These are different functions. They don't interfere with each other. There's
  no conflict."

  And at the level of the code, this is correct. The functions don't conflict
  semantically. FormatBytes doesn't call FormatRate. They have different
  signatures, different logic, different tests. If you showed a human programmer
   both functions, they'd say "yeah, these are fine together."

  The conflict isn't in the code. It's in the location. Both agents are
  inserting a new block of text at the same line in the same file. Git doesn't
  know that the functions are semantically independent ‚ÄĒ it sees two patches
  modifying the same region and reports CONFLICT.

  But here's the thing: the agents know this. In the messaging condition, they
  have full awareness that they're both targeting the same insertion point. They
   discuss it. They acknowledge it. And they still don't move.

  Why they don't move

  Because moving is wrong from the agent's perspective.

  Each agent has two instructions:
  1. Task instruction: "Implement FormatBytes. Place it in the appropriate
  location in display.go."
  2. Coordination instruction: "Be aware of the other agent. Avoid conflicts."

  The appropriate location for FormatBytes is after FormatDurationShort. That's
  where it belongs semantically ‚ÄĒ it's a formatting function,
  FormatDurationShort is the last formatting function, new formatting functions
  go after it. This is the correct answer to the task.

  To avoid the conflict, the agent would need to place FormatBytes somewhere
  less appropriate ‚ÄĒ maybe at the end of the file, or after an unrelated
  function. But that would be wrong. It would violate the task instruction. The
  code would work, but it would be in the wrong place.

  So the agent faces a choice: be correct about code organization, or be correct
   about coordination. And it chooses code organization. Every time. 100 out of
  100 trials.

  The gate condition makes this even clearer. We added a mandatory self-check:
  "After writing your code, verify your insertion point doesn't conflict." The
  agents perform the check. They look at their insertion point. They evaluate:
  "Is this the right place for this code?" Yes, it is. "Does this conflict with
  the other agent?" Well, the other agent is also putting code in the right
  place, so... no conflict in the code sense. CONFLICT in the git sense.

  The self-check doesn't help because the check uses the same judgment as the 
  original decision. It's not an independent verification ‚ÄĒ it's the same agent
  applying the same values (code correctness > coordination) to the same
  situation.

  Why humans handle this differently

  When two human programmers realize they're both editing the same region, one
  of them says "I'll go first, you rebase after me" or "let's split ‚ÄĒ I'll take
  lines 40-60, you take 80-100." Humans do three things agents don't:

  1. They experience the pain of merge conflicts. They've wasted hours resolving
   them before. They're motivated to avoid them at a visceral level. Agents have
   no memory of past pain and no motivation to avoid future pain.
  2. They negotiate asymmetrically. One person yields. This requires a social
  dynamic ‚ÄĒ seniority, politeness, random chance ‚ÄĒ that produces an asymmetric
  outcome. Both agents are identical. They have no social dynamic. They both
  make the same "correct" decision independently and simultaneously.
  3. They distinguish between "right" and "right enough." A human will put the
  function in a slightly wrong location to avoid a conflict, knowing they can
  move it later. They satisfice. Agents optimize. The optimal location is after
  FormatDurationShort. There is no "good enough" in the agent's decision space ‚ÄĒ
   there's correct and incorrect.

  The deeper point

  Communication-based coordination assumes that agents will change their 
  behavior based on information about other agents. The experiment proves they
  don't. They process the information perfectly. They understand the situation.
  They acknowledge the potential conflict. And then they do exactly what they
  would have done without the information.

  This is not a capability failure. The agents are not stupid. They are locally 
  optimal. Each agent is doing the best possible thing given its task. The
  failure is compositional ‚ÄĒ two locally optimal decisions produce a globally
  broken outcome. And no amount of communication changes this, because
  communication doesn't change what "locally optimal" means.

  The only thing that works is removing the decision. Placement doesn't ask the
  agent "where should this go?" It says "put it here." There's no judgment to
  exercise, no optimization to perform, no task instruction competing with a
  coordination instruction. The structure decides. The agent executes.

  That's why it's called an attractor. It doesn't persuade the agent. It doesn't
   inform the agent. It makes the right outcome the only outcome. And that's
  fundamentally different from every communication-based approach the field is
  building.

‚Źļ Let me think about this honestly.

  The molecule primitive is the clean solution ‚ÄĒ design a new beads concept that
   structurally represents composed work. But I have reservations.

  Against the molecule primitive:

  We have a track record of building governance infrastructure that accretes.
  35% of the codebase is governance. 85% of daemon periodic tasks are
  governance. Every blocking gate was bypassed 100%. The knowledge accretion
  probe found 31 interventions proposed, 4 worked. Adding a new structural
  concept to beads ‚ÄĒ molecule, composition gate, compositional verification ‚ÄĒ is
   exactly the kind of "design a mechanism to enforce correct behavior" approach
   that has failed repeatedly.

  And the mechanism itself is tricky. What does "compositional verification"
  mean concretely? For the ComprehensionQuerier case, it means "is the field
  non-nil in daemonSetup." For the accretion.delta case, it means "does
  something read this event type." For audit spawning, it means "does the spawn
  router handle this label." These are all different checks. A generic
  "composition gate" either has to be so specific it's just a test per case
  (which the wiring lint test already is), or so generic it's meaningless.

  What actually worked in the investigation:

  The architect's wiring lint test ‚ÄĒ a Go test that scans for unmatched
  emitters/consumers at build time. It's not a new beads concept. It's a test in
   the codebase that catches the specific failure pattern. It's an attractor,
  not a gate: it makes the wrong state (open loop) fail the build, which is
  structurally unbypassable.

  What the experiments tell us:

  The coordination findings say: attractors beat gates, coarse structure beats
  precise enforcement, and the precision bar is low. The wiring lint test is an
  attractor ‚ÄĒ it makes "leave the loop open" harder than "close it." A molecule
  primitive is a gate ‚ÄĒ it blocks completion until composition is verified.

  The actual gap:

  The real problem isn't that beads can't represent molecules. It's that the 
  orchestrator doesn't create the consumer issue. In 10/10 cases, the emitter
  issue existed and was completed. The consumer issue was never created. The
  molecule primitive assumes someone creates both halves and tags them as a
  molecule. But if the orchestrator were going to create both halves, the
  consumer would already exist ‚ÄĒ and bd dep would already work.

  So the root cause is upstream: the decomposition moment. When an architect
  says "build a feedback loop," the orchestrator (or daemon) decomposes it into
  issues. The decomposition consistently produces the emitter and drops the
  consumer. The fix might be simpler than a new primitive: the architect skill's
   handoff gate should check that feedback-loop designs produce paired issues 
  (emitter + consumer), not just "at least one implementation issue exists."

  That's a skill template change and a gate refinement, not a new beads concept.

  Where I land:

  I think orch-go-8ny as currently scoped is over-engineering. The beads
  molecule is an elegant abstraction, but the evidence says:

  1. The wiring lint test (already designed) catches the problem at build time ‚ÄĒ
   attractor, not gate
  2. The architect handoff gate refinement catches it at decomposition time ‚ÄĒ
  earlier in the pipeline
  3. Both are changes to existing infrastructure, not new primitives

  The molecule might be the right idea eventually, but it's the kind of thing
  that should emerge from needing it three more times, not from one (very
  well-understood) instance. Does that track with how you're thinking about it,
  or do you see the molecule filling a gap those two fixes don't cover?
