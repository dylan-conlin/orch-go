### Self-Describing Artifacts
Generated files and agent-edited artifacts must contain their own operating instructions.
**The test:** "If an agent encounters this file with no memory of how it was created, can they modify it correctly?"
**What this means:**
When artifacts are generated, compiled, or derived from other sources—or when agents create durable state—embed five pieces of information that serve as an "operating manual" for the next consumer:
1. **What is this** - Format marker (e.g., `AUTO-GENERATED by skillc`)
2. **What NOT to do** - Protection warning (e.g., `DO NOT EDIT THIS FILE DIRECTLY`)
3. **Where is source** - Source location (e.g., `Source: .skillc/`)
4. **How to modify** - Rebuild command (e.g., `Run: skillc build`)
5. **When generated** - Timestamp (helps detect staleness)
For agent-edited artifacts (investigations, decisions, issues), this means including metadata (Status, Phase, Next Step) that makes the current state and resumption path explicit.
**What this rejects:**
- "The build process is documented elsewhere" (agent won't find it)
- "Just look at the Makefile" (requires memory to connect artifact → build script)
- "Comments are for humans" (for agents, they're load-bearing infrastructure)
**Key distinction:** For humans, discovering modification instructions is **convenient** (can grep, ask teammates, check history). For agents, it's **necessary** (no persistent memory means no discovery path).
**Examples:**
- `skillc` generated headers in CLAUDE.md
- Protobuf "DO NOT EDIT" warnings with `protoc` command
- Investigation headers (Status, Phase, Next Step)
- Makefile-generated configs with rebuild instructions
**Why this matters:** When consumers have no persistent memory (AI agents), artifacts must carry their own operating manuals. This isn't documentation; it's load-bearing infrastructure. Without self-describing headers, agents will edit generated files directly, breaking the build system and creating conflicts. The five questions above enable correct modification.
**Foundation:** Builds on Session Amnesia principle. Amnesia requires state externalization; self-describing is externalization for generated artifacts.
### Progressive Disclosure
TLDR first. Key sections next. Full details available.
**Why:** Context windows are finite. Attention is limited. Front-load the signal.
**Pattern:** Summary → Key Findings → Details → Appendix
### Surfacing Over Browsing
Bring relevant state to the agent. Don't require navigation.
**Why:** Agents lack persistent memory and spatial intuition. Every file read costs context. Browsing that's cheap for humans is expensive for agents.

```
Phases: design, implementation, validation
Mode: tdd
Validation: smoke-test
```
**This means you will:**
1. Start with Design phase → create design document
2. Move to Implementation phase (TDD mode) → write tests first, then code
3. Finish with Validation phase (smoke-test level) → run tests + manual verification
---
## When to Use Investigation Phase
Investigation phase is OPTIONAL. Include it when:
- ✅ Unfamiliar codebase/subsystem
- ✅ Complex integration points unknown
- ✅ Prior investigation >3 months old
Skip investigation phase when:
- ❌ Architecture well understood
- ❌ Prior investigation exists and current
- ❌ Simple, isolated change
**Alternative:** For significant unknowns, spawn separate investigation agent first,
then spawn feature-impl with `--phases implementation,validation` and reference
to investigation findings.

## Phase Transitions
**After completing each phase:**
Now I see it. The template has:
1. Frontmatter (skill metadata)
2. Intro content (config explanation, deliverables table)
3. Marker section (where phases get inserted)
4. Outro content (phase transitions, completion criteria)
The markers exist because phases are inserted in the middle, not appended at the end.
Why this pattern:
- Common intro/outro shared across phases
- Phases are interchangeable modules
- Order of phases in output matches order in template
Pure concatenation can't do this without splitting intro.md and outro.md and being explicit about order.
So the choice is:
1. Keep markers - Template defines structure, phases plug in
2. Explicit ordering - Split into intro.md, phases/*, outro.md, list all in sources
Option 2 is more verbose but more transparent. Option 1 is more compact but requires understanding the marker system.
What's your preference?

Option 1: Keep markers (template-based)
name: feature-impl
template: SKILL.md.template
phases:
  - investigation
  - design
  - implementation-tdd
Implications:
- Pros:
  - Compact manifest - don't list every file
  - Template controls structure - phases are "pluggable"
  - Familiar to current orch-knowledge users
  
- Cons:
  - Two concepts to understand: templates AND sources
  - Marker syntax is skillc-specific (not standard markdown)
  - Template file is a mix of content and insertion points - harder to read
  - Can't reorder phases without editing template
  - "Where does this content go?" requires reading template to understand
  - Harder to reason about output without running build
---
Option 2: Explicit ordering (pure concatenation)
name: feature-impl
sources:
  - frontmatter.yaml
  - intro.md
  - phases/investigation.md
  - phases/design.md
  - phases/implementation-tdd.md
  - outro.md
Implications:
- Pros:
  - One concept: sources in order
  - Manifest IS the structure - no hidden logic
  - Standard markdown files, no special syntax
  - Reorder by editing manifest, not template
  - Output is predictable: concat(sources)
  - Self-describing: manifest tells you exactly what's in the output
- Cons:
  - Verbose manifest for skills with many phases
  - Must split template into intro/outro (migration work)
  - Shared intro/outro means duplication OR shared files across skills
---
The deeper implication:
Option 1 optimizes for authors (less to type, familiar pattern).
Option 2 optimizes for readers and agents (manifest is the truth, no hidden structure).
Given the self-describing artifacts principle - "artifacts should carry their own operating manual" - Option 2 aligns better. The manifest itself becomes the operating manual for how the skill is assembled.
My take: Option 2. The verbosity is a feature - it's explicit. And skillc is supposed to be the simple tool that "just concatenates files." Adding a template/marker system makes it two tools in one.
