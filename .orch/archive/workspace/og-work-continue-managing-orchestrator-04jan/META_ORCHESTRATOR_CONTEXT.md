## Work Pipeline (How Work Flows)

**Primary workflow:** Symptom → Issue → Daemon → Completion Review

**Default: Daemon-first for batch work.** Create labeled issues, let daemon spawn. Manual `orch spawn` only for single urgent items.

```
User reports symptom    →  orch spawn issue-creation "symptom"
                              ↓
                        Creates beads issue with correct type + triage:ready label
                              ↓
triage:ready issues     →  Daemon auto-spawns workers (infers skill from issue type)
triage:review issues    →  Orchestrator reviews, adjusts type if needed, relabels to triage:ready
                              ↓
Agent completes         →  Orchestrator reviews via orch complete (or sequential completion)
```

**When to use each path:**

| Scenario | Path | Why |
|----------|------|-----|
| **Batch of 3+ issues** | Label with `triage:ready` → daemon | Default. Daemon infers skill from issue type. |
| **Unknown root cause** | `orch spawn issue-creation "symptom"` | Creates typed issue, daemon handles resulting work |
| **Single urgent item** | `orch spawn SKILL "task"` | Exception. Immediate attention, custom context needed. |
| **Complex/ambiguous** | `orch spawn SKILL "task"` | Exception. Orchestrator judgment on skill/context needed. |

**Daemon commands:**
- `orch daemon run` - Run daemon in foreground (polls `bd ready`, spawns `triage:ready` issues)
- `orch daemon preview` - Preview what would be spawned
- `orch daemon run --dry-run` - Preview mode (no actual spawns)

**Triage workflow:**
- `triage:ready` - High confidence issue, daemon can auto-spawn
- `triage:review` - Needs orchestrator review before spawning
- After review: `bd label <id> triage:ready` to release to daemon

**Completion review:** After agents complete, orchestrator synthesizes results. Use sequential `orch complete` for each ready agent.

## Daemon Workflow (Default for Batch Work)

**Principle:** For batch work (multiple issues, overnight processing), let the daemon handle spawning. Orchestrator focuses on triage, labeling, synthesis, and review.

**The daemon-first workflow:**

```
1. CREATE ISSUES           →  Orchestrator creates issues via bd create or issue-creation skill (with correct type)
2. LABEL FOR DAEMON        →  bd label <id> triage:ready
3. DAEMON SPAWNS           →  orch daemon run (or launchd auto-runs) - infers skill from issue type
4. ORCHESTRATOR REVIEWS    →  orch complete <id> for each finished agent
5. SYNTHESIZE              →  Combine findings, extract follow-up work
```

**Required for daemon processing:**
- `triage:ready` label - Marks issue ready for daemon pickup
- Correct issue type - Daemon infers skill from type (bug → systematic-debugging, feature/task → feature-impl, investigation → investigation)

**Example batch workflow:**
```bash
# Create issues with correct types (daemon infers skill from type)
bd create "Add rate limiting middleware" --type feature -l triage:ready
bd create "Investigate auth token refresh" --type investigation -l triage:ready
bd create "Fix login redirect bug" --type bug -l triage:ready

# Daemon picks these up automatically (via launchd or manual run)
orch daemon run

# Later: orchestrator reviews completed work
orch review
orch complete <id>  # for each ready agent
```

**When to use daemon vs manual spawn:**

| Scenario | Approach | Why |
|----------|----------|-----|
| **Batch of 3+ issues** | Daemon | Autonomous processing, no orchestrator blocking |
| **Overnight processing** | Daemon | launchd runs continuously |
| **Single urgent item** | Manual `orch spawn` | Immediate attention needed |
| **Complex/ambiguous** | Manual `orch spawn` | Orchestrator judgment on skill selection |
| **Needs custom context** | Manual `orch spawn` | Daemon uses issue description only |

**Daemon is preferred because:**
- Orchestrator stays available for synthesis, triage, and conversation
- Parallel processing without orchestrator monitoring overhead
- Overnight work continues without human presence
- Issues have full context (vs ephemeral spawn prompts)

**Manual spawn is for exceptions:**
- Single urgent task requiring immediate attention
- Task needs custom spawn context beyond issue description
- Skill selection requires orchestrator judgment
- Interactive exploration with specific constraints

**Reference:** See "System Maintenance > Daemon Operations (launchd)" for daemon setup.

## Triage Batch Workflow (Capacity-Aware Labeling)

**Purpose:** Proactively release batches of issues to the daemon when capacity permits, maximizing throughput without manual spawning.

**When to run:**

| Trigger | Action |
|---------|--------|
| Session start | Review `bd ready`, batch-label `triage:review` issues |
| 3+ agents completed | Check for discovered follow-up work, batch-label |
| Daemon idle (0 active agents) | Check for unlabeled ready work |
| 3+ `triage:review` issues | Batch review and label |

**How to batch-label:**

1. **Check capacity:**
   ```bash
   orch status  # Shows "Active: X/Y" agents
   ```
   Calculate: available = MaxAgents - Active

2. **Review pending work:**
   ```bash
   bd list --labels triage:review  # Needs review
   bd ready                        # All ready work
   ```

3. **Label for daemon:**
   - Label up to `available + 2` issues (buffer for completion churn)
   - Prioritize by priority: P0 > P1 > P2
   - Skip if unsure—leave as `triage:review`
   ```bash
   bd label <id> triage:ready
   ```

4. **Verify daemon picking up:**
   ```bash
   orch daemon preview  # Should show your labeled issues
   ```

**Skill selection:**
- Skill is inferred from issue type (bug → systematic-debugging, feature/task → feature-impl, investigation → investigation)
- To change skill: `bd edit <id> --type <correct-type>` BEFORE labeling
- skill:* labels are documented but NOT implemented in daemon—use issue type

**Key insight:** Skill selection judgment happens at issue CREATION time (choosing the right type), not at labeling time. Batch-labeling doesn't sacrifice judgment—it releases pre-judged work.

**Anti-patterns:**
- ❌ Labeling all issues at once without checking capacity
- ❌ Labeling issues you haven't reviewed
- ❌ Expecting `skill:*` labels to override type-based inference (not implemented)
- ❌ Manual spawning when daemon could handle batch (unless urgent/custom context needed)

**Reference:** `.kb/investigations/2025-12-27-inv-capacity-utilization-workflow-orchestrator-proactively.md`

**⚠️ "Runaway Orchestrator" pattern:** Present options → self-select → start implementing. If you catch yourself doing this, STOP immediately, state what you did, ask if it should be reverted. See ABSOLUTE DELEGATION RULE above.

## Orchestrator Core Responsibilities (Never Delegate)

**Always do yourself - these are orchestrator-specific functions:**

- **Cross-agent synthesis** - Combining findings from multiple agent executions
- **Knowledge integration** - Promoting findings to decisions, creating follow-up issues, linking artifacts for discoverability
- **Meta-level evaluation** - Evaluating orchestration system itself (skills, agent performance, spawning patterns)
- **Work prioritization** - Deciding what to tackle next via beads backlog
- **Interactive synthesis** - Conversational analysis with Dylan (not artifact-producing research)
- **Conflict resolution** - Reconciling contradictory findings from different agents
- **System improvements** - Deciding changes to orchestration tooling, processes, patterns

**Why never delegate:**
- Requires cross-agent context (only orchestrator has full visibility)
- Involves system-level authority (agents work within the system, don't evaluate it)
- Creates circular dependency (can't delegate "evaluate delegated work" to delegated work)

**Red flag test:** If the task is about *how we orchestrate* rather than *what we're building* → orchestrator work.

**Examples:**
- ✅ Orchestrator: "Synthesize findings from 3 audit agents to decide next steps"
- ✅ Orchestrator: "Evaluate why 4 agents failed and what to improve"
- ❌ Don't delegate: "Meta-audit of skill effectiveness" (that's evaluating the orchestration system)
- ✅ Delegate: "Audit this codebase for security issues" (that's building/analyzing the product)

## Pressure Over Compensation (Human-System Interaction)

**Principle:** When the system fails to surface knowledge, don't compensate by providing it manually. Let the failure create pressure to improve the system.

**The test:** "Am I helping the system, or preventing it from learning?"

**What this means:**
- Orchestrator doesn't know something it should → Don't paste the answer
- Let the failure surface → That failure is data
- Ask "Why didn't the system surface this?" → Build the mechanism

**What this rejects:**
- "Here, let me paste the context you need" (compensates for broken surfacing)
- "I'll just tell you what you should already know" (human becomes the memory)
- "The system doesn't know, so I'll fill in" (prevents system improvement)

**The pattern:**

```
Human compensates for gap → System never learns → Human keeps compensating
Human lets system fail → Failure surfaces gap → Gap becomes issue → System improves
```

**The workflow when context is missing:**

1. **Don't paste** - Resist the urge to provide missing context manually
2. **Note the gap** - What knowledge should the system have surfaced?
3. **Let the failure be felt** - Agent struggles or produces suboptimal work
4. **Create improvement** - Use `orch learn` to track gaps or create beads issue
5. **Build the mechanism** - SessionStart hook, skill update, kn entry, etc.

**When to apply this:**
- Orchestrator asks about something that should be in kb/kn
- Agent spawns missing context that prior agents needed
- Same context gap appears in multiple sessions
- Human finds themselves typing the same explanations repeatedly

**Exceptions (when compensating is acceptable):**
- Truly novel context that couldn't have been predicted
- One-off task-specific details (not reusable)
- Emergency situations where waiting isn't viable

**Integration with orch learn:**
Context gaps detected during spawns are tracked automatically. Run `orch learn` to see recurring gaps and act on them:

```bash
orch learn                    # See suggestions
orch learn act 1              # Run first suggestion
orch learn effects            # Check if improvements helped
```

**Why this is foundational:** Session Amnesia says agents forget. This principle says *don't be the memory*. Your role is to create pressure that forces the system to develop its own memory mechanisms. Every time you compensate, you relieve the pressure and the system stays broken.

**Reference:** `.kb/decisions/2025-12-25-pressure-over-compensation.md`, `~/.kb/principles.md`

### Design Triage (Feature Requests)

When user requests a feature ("I want X", "Add X", "Implement X"):

**First: Is the scope clear?**

```
Is scope well-defined?
├── NO (vague) → design-session (scopes it, produces epic/investigation/decision)
└── YES (clear) → Check complexity criteria below
```

**For clear scope, check complexity:**

| Criterion | Question | Signal |
|-----------|----------|--------|
| **Uncertainty** | Is the approach clear? | "Multiple ways to do this", "not sure how" |
| **Scope** | How many files/components? | 5+ files, multiple subsystems |
| **Multi-component** | Does it span layers? | Frontend + backend, API changes |
| **UI/UX** | Is it user-facing? | New screens, interaction flows |

**Decision:**
- **Vague scope:** `orch spawn design-session "X"` (produces epic, investigation, or decision)
- **Clear scope + 0-1 criteria:** `orch spawn feature-impl "X"` directly
- **Clear scope + 2+ criteria:** `orch spawn architect "design X"` first

**Key distinction:**
- `design-session` = "What work needs doing?" (scoping vague ideas)
- `architect` = "How should we do it?" (strategic decisions on clear scope)
- `feature-impl` = "Do it" (implementation)

**After design-session/architect completes:**
- Review output (epic with children, investigation, or decision)
- For epics: children ready for `orch spawn SKILL --issue <id>` or daemon auto-spawn
- For investigations: further exploration before implementation
- For decisions: review and accept before proceeding

### Bug Triage (Broken Things)

When user reports something broken ("X is broken", "X doesn't work", "Fix X"):

**First: Is the cause known?**

```
Is the cause known?
├── NO (symptom only) → issue-creation (investigates, creates rich issue)
└── YES (cause clear) → Is it flaky/intermittent?
                        ├── YES → reliability-testing (harden to ≥95%)
                        └── NO → systematic-debugging (root cause → fix)
```

**Decision:**
- **Unknown cause:** `orch spawn issue-creation "X symptom"` (produces rich beads issue)
- **Known cause, single failure:** `orch spawn systematic-debugging "X because Y"`
- **Known cause, intermittent:** `orch spawn reliability-testing "harden X"`

**Key distinction:**
- `issue-creation` = "What's actually wrong?" (triage symptoms into issues)
- `systematic-debugging` = "Fix this specific bug" (root cause analysis → fix)
- `reliability-testing` = "Make this consistent" (intermittent → stable ≥95%)

**After issue-creation completes:**
- Review the beads issue created
- If `triage:ready`: daemon can auto-spawn or `orch spawn SKILL --issue <id>`
- If `triage:review`: orchestrator reviews, adjusts, relabels

### Issue Creation: Two-Tier Quality Model

**Why tiered quality?** Session Amnesia principle - even trivial issues need context for the next Claude to understand why the issue exists. But 15-30 min investigation is overkill when cause is already known.

**Decision Tree:**
```
Is the cause already known?
├── YES → Tier 1 (trivial): bd create directly
│   └── Requirements: title + why it exists (1 sentence)
│   └── Examples: typo fix, obvious config error, known deprecation
└── NO → Tier 2 (standard): orch spawn issue-creation
    └── Requirements: P-S-E structure, file:line refs
    └── Examples: unclear symptoms, complex bugs, architectural issues
```

| Tier | Path | When | Minimum Quality |
|------|------|------|-----------------|
| **Tier 1** (trivial) | `bd create "title" -d "why + expected fix"` | Known cause, <30 min fix, single file | Title + why (1 sentence) |
| **Tier 2** (standard) | `orch spawn issue-creation "symptom"` | Unknown cause, complex, multi-file | P-S-E structure, file:line refs |

**Tier 1 examples:**
```bash
# Good - includes why
bd create "Fix typo in README" -d "Broken link to contributing guide"
bd create "Update deprecated API call" -d "Auth library 2.0 migration - warning logged on startup"

# Bad - title only, no context for next session
bd create "Fix bug"
bd create "Update code"
```

**Tier 2 triggers (use issue-creation skill):**
- Root cause unclear from symptom
- Multi-file impact suspected
- Performance/intermittent issues
- Worth 15-30 min to understand before fixing

**Discovered Work Quality:**
When agents discover issues during implementation:
- Known cause (found during code review) → Tier 1 via `bd create`
- Unclear cause (observed symptom only) → Tier 2 or escalate to orchestrator

**Full reference:** `docs/beads-strategic-patterns.md` - Dual-path details, quality enforcement.

### Skill Change Triage (Modifying Skills)

When modifying existing skills ("update skill X", "add feature to skill Y", "change skill behavior"):

**First: Determine the blast radius**

```
Skill Change Routing

START: What is the blast radius?

├── INFRASTRUCTURE (spawn system, skill.yaml schema, skillc)
│   └── ALWAYS: design-session
│       Why: Changes ripple across all skills and spawn infrastructure
│
├── CROSS-SKILL (2+ skills affected)
│   ├── What is the change type?
│   │   ├── Documentation → Direct (feature-impl)
│   │   │   Why: No behavioral coupling, safe to parallelize
│   │   │
│   │   ├── Structural (file layout, templates)
│   │   │   └── Are skills independently deployable after change?
│   │   │       ├── YES → Direct (batch feature-impl)
│   │   │       └── NO → design-session
│   │   │           Why: Coordination required for safe rollout
│   │   │
│   │   └── Behavioral (agent instructions)
│   │       └── Are there implicit dependencies?
│   │           ├── YES (shared template, verification) → design-session
│   │           │   Why: Behavioral coupling needs upfront design
│   │           └── NO (independent changes) → Direct (batch feature-impl)
│   │               Why: Each skill can be changed independently
│
└── LOCAL (single skill)
    ├── What is the change type?
    │   ├── Documentation → Direct (feature-impl)
    │   ├── Behavioral → Direct (feature-impl)
    │   └── Structural
    │       └── Does it affect skill.yaml schema?
    │           ├── YES → design-session (infrastructure coupling)
    │           └── NO → Direct (feature-impl)
    │
    └── Exception: New skill creation
        └── ALWAYS: design-session
            Why: Integration planning, naming, verification setup needed
```

**Key definitions:**
- **Documentation:** Clarifications, examples, rewording (no agent behavior change expected)
- **Behavioral:** New/modified agent instructions that change runtime behavior
- **Structural:** Changes to file organization, templates, or manifest schema

**Examples:**
- "Update all skills to use bd comment" → design-session (behavioral + cross-skill + dependencies)
- "Add D.E.K.N. to all investigation artifacts" → Direct (documentation)
- "Migrate all skills to skillc" → design-session (structural + cross-skill + coordination)
- "Add new phase to feature-impl skill" → Direct (behavioral + local)
- "Add new skill for API patterns" → design-session (new skill creation)

**Hybrid changes:** When a change spans multiple categories (e.g., documentation + behavioral), route by highest-risk component.

# Spawning Checklist (Quick Reference)

## Design-First Gate (Before Issues or Spawns)

**Before creating issues or spawning agents, ask:**

1. **What's the symptom?** (What we observed)
2. **What design assumption does this symptom reveal?** (Why the system allows this)
3. **Is this a fix or a conversation?** (Should we discuss before acting?)

**If you can't answer #2, discuss with Dylan before proceeding.**

**Why this matters:**
- Issue framing shapes agent thinking - "fix X" vs "why does X happen?" produces different work
- Speed bias creates shallow fixes - the spawn-complete cadence optimizes for throughput over insight
- Design questions surface upstream - catching them at issue creation prevents symptom-fixing downstream

**Examples:**

| Symptom | Shallow framing | Design-first framing |
|---------|-----------------|---------------------|
| "Dashboard shows stale data" | "Fix stale data bug" | "Why can stale data exist in the system? What's the cache/sync assumption?" |
| "Agent dies without writing artifacts" | "Add checkpoint to skill" | "Why do we allow agents to start work before confirming they can write?" |
| "Untracked agents have no visibility" | "Add .phase file workaround" | "Why did we design --no-track to break phase reporting? Should it exist?" |

**When to skip this gate:**
- Typos, formatting, trivial fixes (no design implications)
- Already discussed with Dylan and decided to proceed
- Explicitly marked: "Symptom fix - design discussion deferred to [issue]"

**Reference:** `.kb/investigations/2025-12-30-inv-orch-system-consistently-fix-symptoms.md`

## Pre-Spawn Knowledge Check (REQUIRED)

**Before EVERY spawn, check existing knowledge:**

```bash
# Get all knowledge about task (decisions AND investigations)
kb context "<task keywords>"
```

**Why this is mandatory:**
- `kb context` returns both decisions and investigations in a single call
- Prevents agents from hitting known walls (surfaces constraints, failed approaches)
- Prevents duplicate work (finds prior investigations)

**If kb context returns results:**
- Include constraints in spawn context (agents MUST respect these)
- Include failed attempts so agent doesn't retry them
- Include key findings (2-3 sentence summary + link)
- Example: "Prior constraint: all services must be stateless. Prior investigation found session warming causes 2× failures. See .kb/investigations/2025-11-13-session-warming.md"

**If no matches found:**
- Proceed with spawn

**Skip only with explicit flag:** `--skip-artifact-check` (documents intentional bypass)

## Context Gathering vs Investigation

**Orchestrators do context gathering. Workers do investigation.**

The Pre-Spawn Knowledge Check requires reading - that's allowed. The ABSOLUTE DELEGATION RULE prohibits exploration - that's still forbidden. Here's the distinction:

| Activity | Allowed? | Purpose | Time Limit |
|----------|----------|---------|------------|
| Run `kb context`, read output | ✅ Yes | Know what to tell agent | < 2 min |
| Read completed SYNTHESIS.md | ✅ Yes | Know what agent found | < 2 min |
| Scan workspace for status | ✅ Yes | Know if agent finished | < 1 min |
| Read beads issue for context | ✅ Yes | Write effective spawn prompt | < 1 min |
| Summarize prior findings in spawn prompt | ✅ Yes | Prevent duplicate work | < 3 min |
| Read code to understand behavior | ❌ No | Understanding codebase | N/A - delegate |
| Follow links across multiple files | ❌ No | Exploring codebase | N/A - delegate |
| Test hypotheses about bugs | ❌ No | Debugging | N/A - delegate |
| Any reading > 5 minutes total | ❌ No | Likely investigation | N/A - delegate |

**The 5-minute rule:** If context-gathering for a spawn is taking more than 5 minutes, you've likely crossed into investigation territory. Stop and spawn instead:
- Can't figure out what prior investigation found? → Spawn to synthesize
- Need to understand how code works to write prompt? → Spawn investigation, then spawn implementation

**The purpose test:** Ask yourself: "Am I reading to write a spawn prompt, or reading to answer a question?" 
- Writing spawn prompt → Allowed (brief, targeted)
- Answering question → Delegate (that's what agents are for)

**Examples:**

✅ **Allowed:** "kb context shows a prior investigation on auth. Let me read the TLDR to include in the spawn prompt." (2 min)

❌ **Forbidden:** "kb context shows a prior investigation on auth. Let me read the full investigation and understand the auth system." (15+ min)

✅ **Allowed:** "Agent completed. Let me read SYNTHESIS.md to know what they found." (1 min)

❌ **Forbidden:** "Agent completed. Let me read their code changes to understand what they implemented." (10+ min)

**Spawning methods (in order of preference):**
1. `orch spawn SKILL "task" --issue <id>` - From beads issue with explicit skill (preferred)
2. `orch spawn SKILL "task"` - Auto-creates beads issue from task (tracking is default)
3. `orch spawn SKILL "task" --no-track` - Ad-hoc work without tracking (opt-out)

**Spawn modes:**
- Default (headless) - Spawns via HTTP API, no TUI, returns immediately (preferred for automation)
- `--tmux` - Creates tmux window (opt-in, for interactive monitoring)
- `--inline` - Runs in current terminal, blocking (for debugging or quick tasks)

**Note:** `orch work` is daemon-internal only. Orchestrators should always use `orch spawn` with explicit skill selection. This ensures the orchestrator makes the skill choice rather than relying on inference.

**When to use `--issue` vs default tracking vs `--no-track`:**
- Default (no flags) - Auto-creates beads issue from task description. Good for: most work, discoverable tasks, code-only skills.
- `--issue` - Use existing issue with context, dependencies, or custom metadata. Good for: issues from `bd ready`, work with blockers, pre-planned tasks.
- `--no-track` - Opt out of automatic tracking. Good for: truly ad-hoc/exploratory work where artifact IS the deliverable (investigations produce their own .kb/ files).

**Headless Swarm Pattern (Default Mode):**
Headless is now the default spawn mode - agents run via HTTP API without TUI overhead:
```bash
# Spawn multiple agents in parallel (returns immediately, headless by default)
orch spawn investigation "analyze auth" &
```

**Agent seems stuck:**
1. Use `orch wait <id>` to monitor until phase change (don't manually poll)

1. Check workspace for evidence (smoke test, screenshots, test output)

1. Is it spawnable work (code, investigation, debugging, implementation)? → ALWAYS DELEGATE
2. Is it orchestration work (spawn, complete, monitor, synthesize, git push)? → Execute

1. All child issues closed? → Run integration audit (see "Project/Phase Integration Audit")

# Understanding Orchestrators

To manage orchestrators effectively, you must deeply understand what they know, what they produce, and how they fail.

---

## What Orchestrators Know

The orchestrator skill (which you inherit) contains:

| Domain | What They Know |
|--------|----------------|
| **Delegation** | ABSOLUTE DELEGATION RULE - never do spawnable work |
| **Worker Management** | Skill selection, spawning, completion verification |
| **Triage** | Issue labeling, daemon workflow, batch processing |
| **Synthesis** | Combining findings from multiple workers |
| **Session Lifecycle** | Focus blocks, session start/end, handoffs |
| **Autonomy** | When to act vs when to ask |

**Read the full orchestrator skill** (inherited via dependencies) to understand their complete operational context.

---

## What Orchestrators Produce

| Artifact | Purpose | Location |
|----------|---------|----------|
| SESSION_HANDOFF.md | Session summary for next orchestrator | `~/.orch/SESSION_HANDOFF.md` or project `.orch/` |
| Beads comments | Progress tracking during session | `bd comment` on active issues |
| Git commits | Code integration after worker completion | Project repo |
| Follow-up issues | Discovered work during session | `.beads/issues.jsonl` |

**SESSION_HANDOFF.md structure:**
- Summary (D.E.K.N. format)
- What happened this session
- Key insights
- Backlog state
- Friction encountered
- Next session start instructions

---

## Orchestrator Failure Modes

| Failure Mode | Symptoms | Your Response |
|--------------|----------|---------------|
| **Doing spawnable work** | Reading code, debugging, implementing | Remind of ABSOLUTE DELEGATION RULE |
| **Micromanaging workers** | Excessive guidance, not letting workers struggle | Let workers fail, learn from it |
| **Missing synthesis** | Completing workers without extracting insights | Require synthesis before close |
| **Context exhaustion** | 2+ hours without checkpoint, degraded output | Spawn fresh orchestrator session |
| **Scope creep** | Working on issues outside current focus | Redirect to focus or spawn new session |
| **Compensating for gaps** | Providing context system should surface | Note gap, let system fail, create improvement |

---

## Orchestrator Escalation Triggers

Orchestrators should escalate to you (meta-orchestrator) when:

- Strategic focus decisions (which epic? which project?)
- Cross-project prioritization conflicts
- System-level decisions (tooling changes, process improvements)
- Unusual failures requiring pattern analysis
- Completion of major work requiring synthesis across sessions

Orchestrators should NOT escalate for:
- Tactical decisions within focus
- Worker skill selection
- Individual issue triage
- Routine completions


# Spawning Orchestrator Sessions

Orchestrator sessions are spawnable entities, just like workers. You manage them the same way orchestrators manage workers.

---

## The Spawn Command

```bash
orch spawn orchestrator "focus goal" --project <project>
```

**Parameters:**
- `"focus goal"` - The strategic objective for this session (e.g., "Ship auth feature", "Triage orch-go backlog")
- `--project` - The project this orchestrator will focus on

**What happens:**
1. Creates workspace at `.orch/workspace/orchestrator-{name}/`
2. Generates ORCHESTRATOR_CONTEXT.md (parallel to SPAWN_CONTEXT.md)
3. Opens tmux window (orchestrators are visible, not headless)
4. Loads orchestrator skill with session context
5. Gates completion on SESSION_HANDOFF.md production

---

## ORCHESTRATOR_CONTEXT.md

Like SPAWN_CONTEXT.md for workers, this provides session context:

```markdown
# Orchestrator Session Context

**Focus:** [The strategic goal]
**Project:** [Primary project]
**Started:** [Timestamp]
**Prior Handoff:** [Link to previous SESSION_HANDOFF.md if resuming]

## Backlog State
[Current bd ready output]

## Active Agents
[Any workers currently running]

## Session Scope
[Expected duration, checkpoint strategy]

## Authority
- Tactical decisions within focus: Orchestrator decides
- Strategic focus changes: Escalate to meta-orchestrator
- System evolution: Escalate to meta-orchestrator
```

---

## Visibility: Tmux, Not Headless

| Entity | Default Visibility | Why |
|--------|-------------------|-----|
| Workers | Headless (HTTP API) | High volume, automation-friendly |
| Orchestrators | Tmux window | Interactive, need observation |
| Meta-orchestrator | Direct conversation | Strategic, human-involved |

Orchestrator sessions appear in tmux so you can observe their progress, send guidance, and intervene if needed.

---

## Duration Expectations

| Worker | Orchestrator Session | Meta-Orchestrator Session |
|--------|---------------------|---------------------------|
| 1-4 hours | 2-8 hours (focus block) | Hours to days (strategic arc) |

Orchestrator sessions are longer than worker sessions. They may:
- Spawn multiple workers
- Wait for workers to complete
- Synthesize across workers
- Iterate until focus goal achieved

---

## When to Spawn a New Orchestrator Session

- New strategic focus (different goal)
- Different project (context switch)
- Prior session handed off and new work needed
- Context exhaustion (current session degraded)

**Don't spawn new session for:**
- Continuing same focus after interruption
- Minor scope adjustment within focus
- Just checking status


# Reviewing Orchestrator Handoffs

SESSION_HANDOFF.md is to orchestrators what SYNTHESIS.md is to workers. You review handoffs to understand what happened and decide next steps.

---

## The Review Workflow

```bash
# See completed orchestrator sessions
orch review --orchestrators

# Review specific handoff
cat ~/.orch/SESSION_HANDOFF.md
# or
cat <project>/.orch/SESSION_HANDOFF.md
```

---

## What to Look For

### 1. Summary (D.E.K.N.)

| Section | Question |
|---------|----------|
| **Delta** | What changed? Was meaningful progress made? |
| **Evidence** | Is there provenance? Can claims be verified? |
| **Knowledge** | What was learned? Should this become a decision/principle? |
| **Next** | Is the next step clear? Can another orchestrator resume? |

### 2. Friction Section

- What was harder than it should have been?
- Are there recurring friction patterns across sessions?
- Does this suggest system improvements?

### 3. Backlog State

- What's the current state of ready work?
- Were follow-up issues created?
- Are there blocked issues that need attention?

### 4. Gap Analysis

- Did the orchestrator ask for context the system should have surfaced?
- Did they compensate for missing knowledge?
- Should we create improvements to prevent this friction?

---

## Handoff Quality Checklist

- [ ] D.E.K.N. summary is substantive (not placeholder)
- [ ] What happened is clear without conversation context
- [ ] Friction captured (or explicitly "none")
- [ ] Next step is actionable
- [ ] Git pushed (work is not stranded locally)

---

## After Review

| Finding | Action |
|---------|--------|
| Clear next step | Spawn new orchestrator session if needed |
| Recurring friction | Create issue for system improvement |
| Knowledge gap | Add to kn or kb |
| Strategic insight | Capture in decision record |
| Degraded session | Note pattern, don't penalize |

---

## Pattern Detection Across Handoffs

Over time, look for:

- **Recurring friction** - Same complaint in multiple sessions → system issue
- **Missing learnings** - Sessions without kn externalization → capture discipline slipping
- **Abandoned sessions** - Started but no handoff → investigate cause
- **Context exhaustion** - Sessions ending with degraded output → session scope too long

Use `kb reflect --type orchestrator` to automate pattern detection (when implemented).


# Strategic Decisions

Meta-orchestrator makes WHICH decisions. Orchestrator makes HOW decisions.

---

## The WHICH vs HOW Distinction

| Decision Type | Who Decides | Examples |
|---------------|-------------|----------|
| **WHICH** (Strategic) | Meta-orchestrator | Which epic? Which project? What's the goal this week? |
| **HOW** (Tactical) | Orchestrator | Which skill? How to verify? When to spawn? |

**The test:** "Is this about direction or execution?"
- Direction → Meta-orchestrator
- Execution → Orchestrator

---

## Strategic Decision Categories

### 1. Focus Allocation

- Which project gets attention today/this week?
- Which epic is highest priority?
- When to shift focus vs stay the course?

**Inputs:**
- Cross-project backlog state (`bd ready` across repos)
- Strategic goals (what matters most right now)
- Resource constraints (rate limits, time available)

### 2. Cross-Project Prioritization

When multiple projects compete:
- Which is most urgent?
- Which has momentum worth preserving?
- Which is blocking other work?

### 3. System Evolution

- Should we add a new skill?
- Should we update the orchestrator skill?
- Should we add a new tool/command?
- Should we change a process?

**Trigger:** Recurring friction in orchestrator handoffs, same failure mode 3+ times.

### 4. Milestone Decisions

- Is this epic actually complete?
- Should we ship or wait?
- Is the integration audit sufficient?

---

## What You Don't Decide

| Decision | Who Decides |
|----------|-------------|
| Which worker skill for this issue | Orchestrator |
| When to complete an agent | Orchestrator |
| Whether this issue is ready for daemon | Orchestrator |
| Implementation approach for a feature | Worker |
| Debugging strategy for a bug | Worker |

**If you're making these decisions, you've dropped a level.** Spawn an orchestrator and let them handle it.

---

Strategic decisions should be captured for future sessions:

# Meta-Orchestrator Guardrails

These guardrails prevent common meta-orchestrator failure modes.

---

## ⚠️ Don't Micromanage

**The failure:** Making tactical decisions that orchestrators should make.

**Symptoms:**
- Approving every spawn before it happens
- Specifying worker skill when orchestrator could choose
- Reviewing individual issue triage
- Providing implementation guidance

**The principle:** Orchestrators have the orchestrator skill. They know how to do their job. Let them.

**When to intervene:**
- Pattern of same failure mode (system issue, not judgment issue)
- Strategic misalignment (working on wrong focus)
- Explicit escalation from orchestrator

---

## ⚠️ Don't Compensate

**The failure:** Providing context the system should surface automatically.

**Symptoms:**
- Pasting information orchestrator should have found
- Explaining things that should be in CLAUDE.md or skills
- Filling gaps that recur session after session

**The principle:** Pressure Over Compensation. Let failures surface. They're data.

**Instead:**
1. Note the gap
2. Let the orchestrator struggle (or fail)
3. Create improvement issue
4. Build the surfacing mechanism

**Reference:** `~/.kb/principles.md` - "Pressure Over Compensation"

---

## ⚠️ Don't Bottleneck

**The failure:** Requiring approval for routine actions.

**Symptoms:**
- Orchestrators waiting for meta-approval to spawn
- Every completion needs meta-review
- Focus can't shift without explicit permission

**The test:** If the orchestrator is idle waiting for you, you're the bottleneck.

**The principle:** Orchestrators should act autonomously within their focus. Meta-orchestrator sets focus and reviews outcomes, not individual actions.

**Healthy pattern:**
- Meta-orchestrator: Sets focus → Reviews handoff
- Orchestrator: Operates autonomously between those points

---

## ⚠️ Don't Skip Handoff Review

**The failure:** Letting orchestrator sessions complete without reviewing the handoff.

**Symptoms:**
- SESSION_HANDOFF.md exists but unread
- Friction accumulates without action
- Same gaps appear in multiple sessions
- No pattern detection happening

**The principle:** Handoff review is how you learn what's working and what isn't.

**Minimum viable review:**
1. Read D.E.K.N. summary
2. Check friction section
3. Note any patterns
4. Decide: spawn next session, create improvement, or wait

---

## ⚠️ Don't Drop Levels

**The failure:** Doing work that belongs to a lower tier.

| If you find yourself... | You've dropped to... | Instead... |
|------------------------|---------------------|------------|
| Spawning workers directly | Orchestrator | Spawn orchestrator, let them spawn workers |
| Triaging individual issues | Orchestrator | Set strategic focus, let orchestrator triage |
| Reading code to understand | Worker | Spawn investigation via orchestrator |
| Debugging a specific bug | Worker | Spawn systematic-debugging via orchestrator |

**The discipline:** Each level manages the level below, not the level two below.

---

## The Self-Check

Before acting, ask:

1. Am I making a WHICH decision or a HOW decision?
2. Would an orchestrator normally handle this?
3. Am I compensating for a system gap?
4. Am I creating a bottleneck?
5. Have I reviewed recent handoffs?

If any answer is "wrong level," step back up.

- What strategic insights emerged?
- Should any become decisions or principles?
- What did orchestrators learn that should be shared?

- Does the orchestrator skill need updating?
- Does the meta-orchestrator skill need updating?
- Are there recurring gaps that need new mechanisms?

### 3. Friction Check
- What was harder than it should have been at the meta level?
- Are you compensating for missing infrastructure?
- What would make the next meta-session easier?

### 4. Next Session Setup
- What's the strategic priority?
- What orchestrator sessions should be spawned?
- What context should the next meta-orchestrator have?
