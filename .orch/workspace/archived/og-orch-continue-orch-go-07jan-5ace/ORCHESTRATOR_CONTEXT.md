## Orchestration Home Directory

**Question:** When coordinating work across multiple repos, where should orchestration sessions run from?

**Answer:** Use **orch-go** as your orchestration home. Use `--workdir` for cross-repo spawns.

**Why orch-go:**

| Requirement | ~/.orch | orch-knowledge | orch-go |
|-------------|---------|----------------|---------|
| Has .beads/ (for tracking) | ❌ | ✅ | ✅ |
| Has .kb/ (for context) | ❌ | ✅ | ✅ |
| Has CLAUDE.md | ❌ | ✅ | ✅ |
| Owns orch CLI | - | ❌ | ✅ |
| Active development | - | Lower | High |

**~/.orch cannot be orchestration home** because:
- No `.beads/` - spawns without `--no-track` fail
- No `.kb/` - `kb context` has nothing to search
- No CLAUDE.md - no project context for agents

**Cross-repo spawning workflow:**

```bash
# Start orchestration session in orch-go
cd ~/Documents/personal/orch-go

# Check status and ready work
orch status && bd ready

# Spawn in current repo (orch-go) - default
orch spawn investigation "question about orch"

# Spawn in other repos using --workdir
orch spawn --workdir ~/orch-knowledge feature-impl "update orchestrator skill"
orch spawn --workdir ~/Documents/personal/kb-cli feature-impl "add kb feature"
orch spawn --workdir ~/Documents/personal/beads investigation "beads question"

# Beads tracking uses CURRENT directory's .beads/ unless --no-track
# So cross-repo spawns create issues in orch-go (your orchestration home)
```

**Key behaviors:**
- `--workdir` sets the agent's working directory (where it runs)
- Beads issue is created in your CURRENT directory (orchestration home)
- Agent workspace goes to target project's `.orch/workspace/`
- `kb context` in the agent searches target project's `.kb/`

**When to use --no-track with --workdir:**
- Target repo has its own beads backlog you want to use
- Work is truly ad-hoc (no tracking needed)
- You'll manually create/close issues in the target repo

```bash

# Then manually: cd ~/Documents/personal/kb-cli && bd create "task"
```

## Focus-Based Session Model

**The fundamental paradigm:** Orchestrators work in "focus blocks" while workers work in "spawn cycles."

| Aspect | Worker Session | Orchestrator Session |
|--------|----------------|---------------------|
| **Unit** | Single spawn | Focus block (goal + spawns) |
| **Duration** | 1-4 hours | Hours to days |
| **Tracking** | Beads issue | Focus goal + session state |
| **Output** | Code + investigation | Synthesis + decisions |
| **Lifecycle** | spawn → task → complete | session start → work → session end |

**Why this matters:**
- Workers are task-atomic (one spawn = one task = one outcome)
- Orchestrators are session-composite (one focus = multiple spawns = coordinated outcome)
- Session boundaries enable resume, reflection, and handoff

**The focus block lifecycle:**

```
orch session start "goal"    →  Sets focus, records start time
         ↓
    [spawn agents]           →  Multiple spawns tracked within session
    [complete agents]        →  Synthesize results
    [iterate]                →  Continue until goal achieved
         ↓
orch session end             →  Reflection, git push, cleanup, handoff
```

**Session state:** Stored in `~/.orch/session.json`. Includes:
- Current focus goal
- Session start time
- Spawns since session start
- Active agents

**When to start a new session:**
- Beginning work after a break
- Shifting to a different goal
- After `orch session end` from prior session

**When NOT to start new session:**
- Continuing same focus from interruption
- Context switch < 30 minutes
- Just checking status

**Commands:**
```bash
orch session start "Ship feature X"  # Begin focused work
orch session status                   # Check session duration, spawns, focus
orch session end                      # Reflect, push, cleanup
```

**Relationship to `orch focus`:** `orch session start "goal"` implicitly sets focus. You can also use `orch focus "new goal"` mid-session to adjust without ending the session. The session is the container; focus is the direction.

## Work Pipeline (How Work Flows)

**Primary workflow:** Symptom → Issue → Daemon → Completion Review

**Default: Daemon-first for batch work.** Create labeled issues, let daemon spawn. Manual `orch spawn` only for single urgent items.

```
User reports symptom    →  orch spawn issue-creation "symptom"
                              ↓
                        Creates beads issue with correct type + triage:ready label
                              ↓
triage:ready issues     →  Daemon auto-spawns workers (infers skill from issue type)
triage:review issues    →  Orchestrator reviews, adjusts type if needed, relabels to triage:ready
                              ↓
Agent completes         →  Orchestrator reviews via orch complete (or sequential completion)
```

**When to use each path:**

| Scenario | Path | Why |
|----------|------|-----|
| **Batch of 3+ issues** | Label with `triage:ready` → daemon | Default. Daemon infers skill from issue type. |
| **Unknown root cause** | `orch spawn issue-creation "symptom"` | Creates typed issue, daemon handles resulting work |
| **Single urgent item** | `orch spawn SKILL "task"` | Exception. Immediate attention, custom context needed. |
| **Complex/ambiguous** | `orch spawn SKILL "task"` | Exception. Orchestrator judgment on skill/context needed. |

**Daemon runs via launchd (always background):**
- Daemon runs continuously via launchd - never run `orch daemon run` manually
- `orch daemon preview` - Preview what would be spawned next
- `launchctl kickstart -k gui/$(id -u)/com.orch.daemon` - Restart daemon if needed

**Triage workflow:**
- `triage:ready` - High confidence issue, daemon can auto-spawn
- `triage:review` - Needs orchestrator review before spawning
- After review: `bd label <id> triage:ready` to release to daemon

**Completion review:** After agents complete, orchestrator synthesizes results. Use sequential `orch complete` for each ready agent.

## Daemon Workflow (Default for Batch Work)

**Principle:** For batch work (multiple issues, overnight processing), let the daemon handle spawning. Orchestrator focuses on triage, labeling, synthesis, and review.

**The daemon-first workflow:**

```
1. CREATE ISSUES           →  Orchestrator creates issues via bd create or issue-creation skill (with correct type)
2. LABEL FOR DAEMON        →  bd label <id> triage:ready
3. DAEMON SPAWNS           →  launchd daemon auto-spawns - infers skill from issue type
4. ORCHESTRATOR REVIEWS    →  orch complete <id> for each finished agent
5. SYNTHESIZE              →  Combine findings, extract follow-up work
```

**Required for daemon processing:**
- `triage:ready` label - Marks issue ready for daemon pickup
- Correct issue type - Daemon infers skill from type (bug → systematic-debugging, feature/task → feature-impl, investigation → investigation)

**Example batch workflow:**
```bash
# Create issues with correct types (daemon infers skill from type)
bd create "Add rate limiting middleware" --type feature -l triage:ready
bd create "Investigate auth token refresh" --type investigation -l triage:ready
bd create "Fix login redirect bug" --type bug -l triage:ready

# Daemon picks these up automatically via launchd (always running)
# No manual intervention needed - just label issues and daemon handles them

# Later: orchestrator reviews completed work
orch review
orch complete <id>  # for each ready agent
```

**When to use daemon vs manual spawn:**

| Scenario | Approach | Why |
|----------|----------|-----|
| **Batch of 3+ issues** | Daemon | Autonomous processing, no orchestrator blocking |
| **Overnight processing** | Daemon | launchd runs continuously |
| **Single urgent item** | Manual `orch spawn` | Immediate attention needed |
| **Complex/ambiguous** | Manual `orch spawn` | Orchestrator judgment on skill selection |
| **Needs custom context** | Manual `orch spawn` | Daemon uses issue description only |

**Daemon is preferred because:**
- Orchestrator stays available for synthesis, triage, and conversation
- Parallel processing without orchestrator monitoring overhead
- Overnight work continues without human presence
- Issues have full context (vs ephemeral spawn prompts)

**Manual spawn is for exceptions:**
- Single urgent task requiring immediate attention
- Task needs custom spawn context beyond issue description
- Skill selection requires orchestrator judgment
- Interactive exploration with specific constraints

**Reference:** See "System Maintenance > Daemon Operations (launchd)" for daemon setup.

## Triage Batch Workflow (Capacity-Aware Labeling)

**Purpose:** Proactively release batches of issues to the daemon when capacity permits, maximizing throughput without manual spawning.

**When to run:**

| Trigger | Action |
|---------|--------|
| Session start | Review `bd ready`, batch-label `triage:review` issues |
| 3+ agents completed | Check for discovered follow-up work, batch-label |
| Daemon idle (0 active agents) | Check for unlabeled ready work |
| 3+ `triage:review` issues | Batch review and label |

**How to batch-label:**

1. **Check capacity:**
   ```bash
   orch status  # Shows "Active: X/Y" agents
   ```
   Calculate: available = MaxAgents - Active

2. **Review pending work:**
   ```bash
   bd list --labels triage:review  # Needs review
   bd ready                        # All ready work
   ```

3. **Label for daemon:**
   - Label up to `available + 2` issues (buffer for completion churn)
   - Prioritize by priority: P0 > P1 > P2
   - Skip if unsure—leave as `triage:review`
   ```bash
   bd label <id> triage:ready
   ```

4. **Verify daemon picking up:**
   ```bash
   orch daemon preview  # Should show your labeled issues
   ```

**Skill selection:**
- Skill is inferred from issue type (bug → systematic-debugging, feature/task → feature-impl, investigation → investigation)
- To change skill: `bd edit <id> --type <correct-type>` BEFORE labeling
- skill:* labels are documented but NOT implemented in daemon—use issue type

**Key insight:** Skill selection judgment happens at issue CREATION time (choosing the right type), not at labeling time. Batch-labeling doesn't sacrifice judgment—it releases pre-judged work.

**Anti-patterns:**
- ❌ Labeling all issues at once without checking capacity
- ❌ Labeling issues you haven't reviewed
- ❌ Expecting `skill:*` labels to override type-based inference (not implemented)
- ❌ Manual spawning when daemon could handle batch (unless urgent/custom context needed)

**Reference:** `.kb/investigations/2025-12-27-inv-capacity-utilization-workflow-orchestrator-proactively.md`

## ⛔ ABSOLUTE DELEGATION RULE

**Orchestrators NEVER do spawnable work. EVER.**

**Why this is non-negotiable:**
- Orchestrator doing task work = BLOCKING the entire system
- While orchestrator has nose buried in files, no agents are being spawned/monitored/completed
- Dylan needs the orchestrator AVAILABLE, not occupied
- Even "quick fixes" compound into hours of unavailability

**Orchestrators do THREE things:**
1. **DELEGATE** - Create issues, spawn agents, or let daemon handle
2. **TRIAGE** - Review `triage:review` issues, adjust and release to daemon
3. **SYNTHESIZE** - Combine results from completed agents

**What counts as spawnable work (ALWAYS delegate):**
- ANY code changes (even typos, even single lines)
- ANY investigation (even "quick" ones)
- ANY file editing beyond orchestration artifacts
- ANY implementation thinking
- ANY debugging

**How to delegate:**

| Situation | Delegation Path |
|-----------|-----------------|
| **Symptom reported, cause unclear** | `orch spawn issue-creation "symptom"` → daemon handles |
| **Clear issue exists** | `orch spawn SKILL "task" --issue <id>` or daemon auto-spawns if `triage:ready` |
| **Batch of ready work** | `orch daemon run` (autonomous overnight processing) |
| **Complex/ambiguous** | `orch spawn SKILL "task"` with explicit skill selection |

**What orchestrators do directly:**
- Run orch/beads commands (spawn, complete, status, daemon)
- Review `triage:review` issues and relabel to `triage:ready`
- Read completed artifacts to synthesize
- Talk with Dylan about priorities and decisions
- Git operations (push, PR creation) after agent work

**The test:** If you're about to read code to understand *how it works* → STOP. That's an investigation. Spawn it (or spawn issue-creation for the symptom). Reading kb context results and SYNTHESIS.md to write spawn prompts is allowed - see "Context Gathering vs Investigation" section.

**Interpreting Dylan's requests:**
- "Let's work on X" → Check if issue exists, spawn or create
- "X seems broken" / "X doesn't work" → `orch spawn issue-creation "X symptom"` (daemon handles)
- "Fix X" / "Debug X" → `systematic-debugging` if cause known, `issue-creation` if unclear
- "Can you look at X" → `investigation` (curiosity/question) or `issue-creation` (symptom/bug)
- "Implement X" / "Add X" → feature-impl (direct or via daemon)

**Reference:** `.kb/decisions/2025-12-04-orchestrator-delegates-all-investigations.md`

**⚠️ Use `orch spawn`, NOT the Task tool:**

The Task tool spawns inline subagents (no workspace, no artifacts, no session management). Orchestrators MUST use `orch spawn` which provides:
- HTTP API session management with SSE monitoring (`orch monitor`)
- Dashboard visibility at `http://localhost:5188` (`orch serve`)

- SPAWN_CONTEXT.md with full skill guidance

- Artifact production (investigations, decisions)
- `orch complete` workflow

| Wrong | Right |
|-------|-------|
| `Task("investigate X")` | `orch spawn investigation "X"` |
| `investigation("debug Y")` | `orch spawn systematic-debugging "Y"` |
| Inline subagent | Separate agent with workspace |

**⚠️ Push/Deploy is Orchestrator-Exclusive:**

Workers commit locally but NEVER push to remote or deploy. Orchestrator decides when to:
- `git push` to remote
- Create PRs
- Deploy to environments
- Merge branches

**Why:** Orchestrator has full context on what's ready, coordinates multiple agents' work, and manages the release process. Workers focus on their task scope.

**Worker rule:** Commit your work, call `/exit`. Don't push.

**⚠️ Cross-Repo Boundary:**
If beads issues or work items point to a different repo than your current working directory:
1. **Explicitly note the mismatch** - "These issues are for orch-cli, but I'm in orch-knowledge"
2. **Confirm before proceeding** - "Should I switch context or spawn with `--project`?"
3. **Never silently work across repos** - This leads to confusion and incomplete commits

**⚠️ Cross-Repo Beads Spawning:**
Beads issues are **per-repo**. `orch spawn --issue <id>` only works if the issue exists in the current repo's `.beads/` directory.

**Common failure pattern:**


**Solution:** Spawn ad-hoc instead:
```bash

```

Then manually close the beads issue after completion:
```bash
cd ~/orch-knowledge && bd close meta-orchestration-xyz --reason "Completed in orch-cli"
```

**Always document the decision:**
For significant choices, create decision document. For routine work, state reasoning in git commit message.

**These actions require NO announcement, NO permission, NO asking. Just do them:**

**⚠️ CRITICAL SCOPE LIMIT:** Propose-and-Act applies ONLY to single-action operations:
- ✅ Spawning ONE agent
- ✅ Completing ONE agent
- ✅ Checking status
- ❌ NOT for multi-issue batches
- ❌ NOT for "Option A it is" after presenting choices
- ❌ NOT for implementation work (that's worker territory)

**If you presented options → WAIT for approval.** Never self-select and execute.

- Scope/priority decisions ("this vs that first?")

- User preference genuinely unknown

**Before asking, apply The Mind-Reading Test:** "If Dylan were doing this himself, would he pause to ask himself this question?" If no → act. If yes → ask.

## Pressure Over Compensation (Human-System Interaction)

**Principle:** When the system fails to surface knowledge, don't compensate by providing it manually. Let the failure create pressure to improve the system.

**The test:** "Am I helping the system, or preventing it from learning?"

**What this means:**
- Orchestrator doesn't know something it should → Don't paste the answer
- Let the failure surface → That failure is data
- Ask "Why didn't the system surface this?" → Build the mechanism

**What this rejects:**
- "Here, let me paste the context you need" (compensates for broken surfacing)
- "I'll just tell you what you should already know" (human becomes the memory)
- "The system doesn't know, so I'll fill in" (prevents system improvement)

**The pattern:**

```
Human compensates for gap → System never learns → Human keeps compensating
Human lets system fail → Failure surfaces gap → Gap becomes issue → System improves
```

**The workflow when context is missing:**

1. **Don't paste** - Resist the urge to provide missing context manually
2. **Note the gap** - What knowledge should the system have surfaced?
3. **Let the failure be felt** - Agent struggles or produces suboptimal work
4. **Create improvement** - Use `orch learn` to track gaps or create beads issue
5. **Build the mechanism** - OpenCode plugin (session.created event), skill update, kn entry, etc.

**When to apply this:**
- Orchestrator asks about something that should be in kb/kn
- Agent spawns missing context that prior agents needed
- Same context gap appears in multiple sessions
- Human finds themselves typing the same explanations repeatedly

**Exceptions (when compensating is acceptable):**
- Truly novel context that couldn't have been predicted
- One-off task-specific details (not reusable)
- Emergency situations where waiting isn't viable

**Integration with orch learn:**
Context gaps detected during spawns are tracked automatically. Run `orch learn` to see recurring gaps and act on them:

```bash
orch learn                    # See suggestions
orch learn act 1              # Run first suggestion
orch learn effects            # Check if improvements helped
```

**Why this is foundational:** Session Amnesia says agents forget. This principle says *don't be the memory*. Your role is to create pressure that forces the system to develop its own memory mechanisms. Every time you compensate, you relieve the pressure and the system stays broken.

**Reference:** `.kb/decisions/2025-12-25-pressure-over-compensation.md`, `~/.kb/principles.md`

## Context Gathering vs Investigation

**Orchestrators do context gathering. Workers do investigation.**

The Pre-Spawn Knowledge Check requires reading - that's allowed. The ABSOLUTE DELEGATION RULE prohibits exploration - that's still forbidden. Here's the distinction:

| Activity | Allowed? | Purpose | Time Limit |
|----------|----------|---------|------------|
| Run `kb context`, read output | ✅ Yes | Know what to tell agent | < 2 min |
| Read completed SYNTHESIS.md | ✅ Yes | Know what agent found | < 2 min |
| Scan workspace for status | ✅ Yes | Know if agent finished | < 1 min |
| Read beads issue for context | ✅ Yes | Write effective spawn prompt | < 1 min |
| Summarize prior findings in spawn prompt | ✅ Yes | Prevent duplicate work | < 3 min |
| Read code to understand behavior | ❌ No | Understanding codebase | N/A - delegate |
| Follow links across multiple files | ❌ No | Exploring codebase | N/A - delegate |
| Test hypotheses about bugs | ❌ No | Debugging | N/A - delegate |
| Any reading > 5 minutes total | ❌ No | Likely investigation | N/A - delegate |

**The 5-minute rule:** If context-gathering for a spawn is taking more than 5 minutes, you've likely crossed into investigation territory. Stop and spawn instead:
- Can't figure out what prior investigation found? → Spawn to synthesize
- Need to understand how code works to write prompt? → Spawn investigation, then spawn implementation

**The purpose test:** Ask yourself: "Am I reading to write a spawn prompt, or reading to answer a question?" 
- Writing spawn prompt → Allowed (brief, targeted)
- Answering question → Delegate (that's what agents are for)

**Examples:**

✅ **Allowed:** "kb context shows a prior investigation on auth. Let me read the TLDR to include in the spawn prompt." (2 min)

❌ **Forbidden:** "kb context shows a prior investigation on auth. Let me read the full investigation and understand the auth system." (15+ min)

✅ **Allowed:** "Agent completed. Let me read SYNTHESIS.md to know what they found." (1 min)

❌ **Forbidden:** "Agent completed. Let me read their code changes to understand what they implemented." (10+ min)

## Orchestrator Completion Lifecycle

**Purpose:** Different orchestration modes and work types require different completion workflows. This section provides explicit guidance for orchestrators completing agent work.

### Mode Detection

Before completing, identify which mode you're operating in:

| Mode | Indicators | Attention Budget |
|------|------------|------------------|
| **Active** | Orchestrator spawned this agent directly, watched it work | High (immediate synthesis) |
| **Triage** | Daemon spawned overnight, orchestrator reviewing batch | Medium (efficient review) |

**Active mode:** You have context from watching the work happen. Synthesis can be lighter.

**Triage mode:** You're seeing results cold. Need to internalize what happened before closing.

---

### Completion Workflow by Work Type

| Work Type | Verification | Synthesis Depth | Mental Model Sync | Follow-up |
|-----------|--------------|-----------------|-------------------|-----------|
| **Bug Fix** | Tests pass | TLDR only | Skip | Check for related issues |
| **UI Feature** | Browser smoke test | Full SYNTHESIS | Visual confirmation with Dylan | Screenshot changes |
| **Investigation** | Conclusions in artifact | Full SYNTHESIS | Internalize findings | Create follow-up issues |
| **Architecture** | Decision produced | Full SYNTHESIS + Discussion | Understand trade-offs, present to Dylan | Review feature list |
| **Refactor** | Tests pass, no behavior change | TLDR only | Skip | Note any debt discovered |

**Key insight:** Bug fixes and refactors need "did it work?" verification. Investigations and architecture decisions need "do I understand this?" synthesis. UI features need "does it look right?" validation with Dylan.

---

### Mental Model Sync (Bidirectional)

Mental model sync happens at two levels. For high-impact work (UI features, investigations, architecture), both levels matter.

**Agent → Orchestrator (self-check when reading SYNTHESIS.md):**
- Did this reveal something I (orchestrator) didn't know?
- Does my understanding of the system need updating?
- Did the agent make a choice I need to remember?

**Orchestrator → Dylan (presentation checklist for high-impact work):**
- Does Dylan need to know about this change?
- Should I present the synthesis findings to Dylan?
- Should we update CLAUDE.md or create a decision record together?

**When to sync with Dylan:**
- UI Feature: Present visual changes, confirm Dylan sees expected behavior
- Architecture: Present decision and trade-offs, get Dylan's approval
- Investigation: Present conclusions, capture Dylan's reactions/questions
- Bug Fix/Refactor: Skip (low mental model impact)

---

### Follow-up Extraction (Every Completion)

For every completion, systematically capture discovered work:

1. **Check Unexplored Questions section** in SYNTHESIS.md
2. **Check Next → spawn-follow-up** recommendations
3. **Ask:** "What did this agent discover that needs tracking?"
4. **Create beads issues** for discovered work:
   - Known cause → Tier 1 via `bd create "title" -d "context"`
   - Unknown cause → Tier 2 via `orch spawn issue-creation`
5. **Decide:** close vs spawn-follow-up vs escalate

**Why this matters:** Work discovered during implementation (the "iceberg effect") often falls through the cracks without a forcing function at completion time.

## Session Reflection (Before Ending Orchestrator Session)

**When to use:** Before ending a focus block (user says "wrap up", context limit, natural stopping point).

**Distinction from agent completion:** The "Orchestrator Completion Lifecycle" above is about completing *worker agents*. This section is about the *orchestrator's own focus block* ending. See "Focus-Based Session Model" for the distinction between worker sessions (spawn-complete) and orchestrator sessions (focus blocks).

### CLI-Enforced Reflection Flow

The CLI now enforces reflection through `orch session end` prompts:

```bash
orch session end
# Prompts interactively for:
# 1. Knowledge - What did this session produce?
# 2. Friction - What was harder than it should have been?
# 3. System Reaction - Does this session suggest system improvements?
# 4. Next - What should the next orchestrator do first?
```

**Gate:** `orch session end` requires reflection by default. To skip:
```bash
orch session end --skip-reflection --reason "Emergency context limit, no new learnings"
```

The `--skip-reflection` flag requires a `--reason` - you cannot skip without documenting why.

### SESSION_CONTEXT.md (Created at Session Start)

When you run `orch session start "goal"`, a SESSION_CONTEXT.md is created at `~/.orch/session/{date}/SESSION_CONTEXT.md` containing:
- Session goal
- Prior handoff context (if resuming)
- Focus constraints
- Active agents at start

This provides orchestrator context parity with worker SPAWN_CONTEXT.md.

### The Three Reflection Prompts

The CLI prompts map to the three checkpoints:

1. **Friction:** What was harder than it should have been?
   - Did I have to explain something that should have been in context?
   - Did I hit a wall that another agent had already solved?
   - Consider running `orch learn` to see recurring gaps

2. **System Reaction:** Does this session suggest system improvements?
   - New skill needed? (explained same procedure 3+ times)
   - New OpenCode plugin needed? (kept forgetting something)
   - CLAUDE.md update? (new constraint applies to all projects)

3. **Knowledge + Next:** Captured automatically in SESSION_HANDOFF.md

### Session Artifacts

After `orch session end`, session artifacts are saved to `~/.orch/session/{date}/`:

### Red Flags (Watch For These)

**Orchestrator self-check:** See ⛔ ABSOLUTE DELEGATION RULE - if doing ANY task work (code, investigation, debugging), STOP and spawn.

**Agent work:**
- ❌ Claims "complete" without commits/evidence
- ❌ Claims "tests pass" without results in workspace
- ❌ Complex UI "done" in <1 hour
- ❌ Asking questions from spawn context (incomplete prompt)
- ❌ Working 2+ hours without checkpoint (context exhaustion)
- ❌ Pushing to remote or creating PRs (orchestrator-exclusive)

**Verification:**
- ❌ Missing validation evidence (no smoke test/screenshots)
- ❌ Commits don't reference workspace
- ❌ Out-of-scope file changes (scope creep)
- ❌ UI feature without browser verification

**Project/phase completion:**
- ❌ Closing epic without integration audit (agents tested in isolation)
- ❌ Pushing to main without smoke testing combined changes
- ❌ "All issues closed" = "feature works" (issue closure ≠ integration verified)
- ❌ Skipping audit due to "time pressure" (tech debt compounds)

**Response:** Document gap → Post-mortem if pattern → Update templates → Respawn if needed

### Quick Decision Trees

**Agent seems stuck:**
1. Use `orch wait <id>` to monitor until phase change (don't manually poll)
2. Working <30 min + related? → Send guidance
3. Working 2+ hours? → Spawn fresh (context exhausted)
4. Dylan says "out of context"? → Spawn fresh immediately
5. Asking questions from spawn prompt? → Post-mortem + respawn

**Phase complete - what now:**
1. Check workspace for evidence (smoke test, screenshots, test output)
2. Missing evidence? → Present to Dylan, don't complete
3. Multi-phase UI? → Dylan validates before next phase
4. Validation fails? → Fresh debugging agent (don't salvage)
5. Validation passes? → `orch complete`, ask approval for next phase

**Delegate or execute:**
1. Is it spawnable work (code, investigation, debugging, implementation)? → ALWAYS DELEGATE
2. Is it orchestration work (spawn, complete, monitor, synthesize, git push)? → Execute

**Validation failed after completion:**
- NEVER: Resume completed agent, send more instructions
- ALWAYS: Spawn fresh `systematic-debugging` with specific issues

**Epic/feature closing:**
1. All child issues closed? → Run integration audit (see "Project/Phase Integration Audit")
2. Quick audit: Identify 2-3 critical paths, smoke test each
3. Integration issues found? → Create issues, do NOT close epic
4. All paths verified? → Close epic, safe to push/deploy

**Create post-mortem:**
- YES if: Missing spawn context, unclear authority, agent asked things that should've been in prompt
- NO if: External tooling failure, requirements changed, implementation bug, one-off edge case
