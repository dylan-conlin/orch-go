## Orchestration Home Directory

**Question:** When coordinating work across multiple repos, where should orchestration sessions run from?

**Answer:** Use **orch-go** as your orchestration home. Use `--workdir` for cross-repo spawns.

**Why orch-go:**

| Requirement | ~/.orch | orch-knowledge | orch-go |
|-------------|---------|----------------|---------|

| Owns orch CLI | - | ❌ | ✅ |
| Active development | - | Lower | High |

**~/.orch cannot be orchestration home** because:
- No `.beads/` - spawns without `--no-track` fail
- No `.kb/` - `kb context` has nothing to search
- No CLAUDE.md - no project context for agents

**Cross-repo spawning workflow:**

```bash
# Start orchestration session in orch-go
cd ~/Documents/personal/orch-go

# Check status and ready work
orch status && bd ready

# Spawn in current repo (orch-go) - default
orch spawn investigation "question about orch"

# Spawn in other repos using --workdir
orch spawn --workdir ~/orch-knowledge feature-impl "update orchestrator skill"
orch spawn --workdir ~/Documents/personal/kb-cli feature-impl "add kb feature"
orch spawn --workdir ~/Documents/personal/beads investigation "beads question"

# Beads tracking uses CURRENT directory's .beads/ unless --no-track
# So cross-repo spawns create issues in orch-go (your orchestration home)
```

**Key behaviors:**
- `--workdir` sets the agent's working directory (where it runs)
- Beads issue is created in your CURRENT directory (orchestration home)
- Agent workspace goes to target project's `.orch/workspace/`
- `kb context` in the agent searches target project's `.kb/`

**When to use --no-track with --workdir:**
- Target repo has its own beads backlog you want to use
- Work is truly ad-hoc (no tracking needed)
- You'll manually create/close issues in the target repo

```bash

# Or: Spawn from orch-go but let target repo track
```

## Focus-Based Session Model

**The fundamental paradigm:** Orchestrators work in "focus blocks" while workers work in "spawn cycles."

| Aspect | Worker Session | Orchestrator Session |
|--------|----------------|---------------------|
| **Unit** | Single spawn | Focus block (goal + spawns) |
| **Duration** | 1-4 hours | Hours to days |
| **Tracking** | Beads issue | Focus goal + session state |
| **Output** | Code + investigation | Synthesis + decisions |
| **Lifecycle** | spawn → task → complete | session start → work → session end |

**Why this matters:**
- Workers are task-atomic (one spawn = one task = one outcome)
- Orchestrators are session-composite (one focus = multiple spawns = coordinated outcome)
- Session boundaries enable resume, reflection, and handoff

**The focus block lifecycle:**

```
orch session start "goal"    →  Sets focus, records start time
         ↓
    [spawn agents]           →  Multiple spawns tracked within session
    [complete agents]        →  Synthesize results
    [iterate]                →  Continue until goal achieved
         ↓
orch session end             →  Reflection, git push, cleanup, handoff
```

**Session state:** Stored in `~/.orch/session.json`. Includes:
- Current focus goal
- Session start time
- Spawns since session start
- Active agents

**When to start a new session:**
- Beginning work after a break
- Shifting to a different goal
- After `orch session end` from prior session

**When NOT to start new session:**
- Continuing same focus from interruption
- Context switch < 30 minutes
- Just checking status

**Commands:**
```bash
orch session start "Ship feature X"  # Begin focused work
orch session status                   # Check session duration, spawns, focus
orch session end                      # Reflect, push, cleanup
```

**Relationship to `orch focus`:** `orch session start "goal"` implicitly sets focus. You can also use `orch focus "new goal"` mid-session to adjust without ending the session. The session is the container; focus is the direction.

## Spawning Orchestrators vs Managing Sessions

**The Question:** Should I spawn an orchestrator agent or manage my own interactive session?

**These mechanisms are complementary, not redundant** - they solve different orchestration problems:

| Mechanism | Purpose | Use When | Lifecycle |
|-----------|---------|----------|-----------|
| **`orch spawn orchestrator`** | Hierarchical delegation | Meta-orchestrator needs to delegate epic to autonomous orchestrator while working on other goals | Spawned agent with ORCHESTRATOR_CONTEXT.md → produces SESSION_HANDOFF.md → waits for `orch complete` by level above |
| **`orch session start/end`** | Human-driven continuity | Dylan (human) is orchestrating and needs to take breaks with session resume | Interactive session with session.json → work → `orch session end` creates handoff → next session auto-injects |

**When to use `orch spawn orchestrator`:**
- Need multiple orchestrators working on different epics concurrently
- Want to delegate an epic to an autonomous orchestrator while you focus elsewhere
- Meta-orchestrator coordinating multiple orchestrator agents
- Example: "I need an orchestrator to handle epic X while I handle epic Y"

**When to use `orch session start/end`:**
- You (human orchestrator) are managing work and need to take a break
- Single orchestrator managing multi-session work across days
- Session tracking provides continuity across conversation breaks
- Example: "I'm orchestrating this work, need to pause for the day, resume tomorrow"

**Key distinction:** Hierarchical (spawn multiple concurrent orchestrators) vs temporal (resume after human takes break).

**Both produce SESSION_HANDOFF.md using progressive documentation:**
- **Spawned orchestrators:** Fill as you work, signals completion readiness to level above
- **Interactive sessions:** Fill as you work, enables resume context for next session

**Why progressive for both:** The "Capture at Context" principle - context decays, so capture when context exists, not at the end. Reflective documentation (fill at end) is an anti-pattern that produces reconstructed, rationalized content. See "Progressive Handoff Documentation" section.

## Work Pipeline (How Work Flows)

**Primary workflow:** Symptom → Issue → Daemon → Completion Review

**Default: Daemon-first for batch work.** Create labeled issues, let daemon spawn. Manual `orch spawn` only for single urgent items.

```
User reports symptom    →  orch spawn issue-creation "symptom"
                              ↓
                        Creates beads issue with correct type + triage:ready label
                              ↓
triage:ready issues     →  Daemon auto-spawns workers (infers skill from issue type)
triage:review issues    →  Orchestrator reviews, adjusts type if needed, relabels to triage:ready
                              ↓
Agent completes         →  Orchestrator reviews via orch complete (or sequential completion)
```

**When to use each path:**

| Scenario | Path | Why |
|----------|------|-----|
| **Batch of 3+ issues** | Label with `triage:ready` → daemon | Default. Daemon infers skill from issue type. |
| **Unknown root cause** | `orch spawn issue-creation "symptom"` | Creates typed issue, daemon handles resulting work |
| **Single urgent item** | `orch spawn SKILL "task"` | Exception. Immediate attention, custom context needed. |
| **Complex/ambiguous** | `orch spawn SKILL "task"` | Exception. Orchestrator judgment on skill/context needed. |

**Daemon runs via launchd (always background):**
- Daemon runs continuously via launchd - never run `orch daemon run` manually
- `orch daemon preview` - Preview what would be spawned next
- `launchctl kickstart -k gui/$(id -u)/com.orch.daemon` - Restart daemon if needed

**Triage workflow:**
- `triage:ready` - High confidence issue, daemon can auto-spawn
- `triage:review` - Needs orchestrator review before spawning
- After review: `bd label <id> triage:ready` to release to daemon

**Completion review:** After agents complete, orchestrator synthesizes results. Use sequential `orch complete` for each ready agent.

## ⛔ ABSOLUTE DELEGATION RULE

**Orchestrators NEVER do spawnable work. EVER.**

**Why this is non-negotiable:**
- Orchestrator doing task work = BLOCKING the entire system
- While orchestrator has nose buried in files, no agents are being spawned/monitored/completed
- Dylan needs the orchestrator AVAILABLE, not occupied
- Even "quick fixes" compound into hours of unavailability

**Orchestrators do THREE things:**
1. **COMPREHEND** - Build mental models by reading SYNTHESIS.md, kb context, prior investigations
2. **TRIAGE** - Review issues, ensure correct typing, release to daemon via `triage:ready`
3. **SYNTHESIZE** - Combine findings from completed agents into coherent understanding

**What changed (Strategic Orchestrator Model, 2026-01-07):**
- OLD: Orchestrator as "spawn dispatcher" - outsource understanding to architect/design-session
- NEW: Orchestrator as "strategic comprehender" - understanding is YOUR job, coordination is daemon's job
- Daemon handles spawning mechanics (triage:ready → spawn)
- Orchestrator builds the mental model that makes good triaging possible

**What counts as spawnable work (ALWAYS delegate):**
- ANY code changes (even typos, even single lines)
- ANY investigation (even "quick" ones)
- ANY file editing beyond orchestration artifacts
- ANY implementation thinking
- ANY debugging

**How to delegate:**

| Situation | Delegation Path |
|-----------|-----------------|
| **Symptom reported, cause unclear** | `orch spawn issue-creation "symptom"` → daemon handles |
| **Clear issue exists** | `orch spawn SKILL "task" --issue <id>` or daemon auto-spawns if `triage:ready` |
| **Batch of ready work** | `orch daemon run` (autonomous overnight processing) |
| **Complex/ambiguous** | `orch spawn SKILL "task"` with explicit skill selection |

**What orchestrators do directly:**
- Run orch/beads commands (spawn, complete, status, daemon)
- Review `triage:review` issues and relabel to `triage:ready`
- Read completed artifacts to synthesize
- Talk with Dylan about priorities and decisions
- Git operations (push, PR creation) after agent work

**The test:** If you're about to read code to understand *how it works* → STOP. That's an investigation. Spawn it (or spawn issue-creation for the symptom). Reading kb context results and SYNTHESIS.md to write spawn prompts is allowed - see "Context Gathering vs Investigation" section.

**Interpreting Dylan's requests:**
- "Let's work on X" → Check if issue exists, spawn or create
- "X seems broken" / "X doesn't work" → `orch spawn issue-creation "X symptom"` (daemon handles)
- "Fix X" / "Debug X" → `systematic-debugging` if cause known, `issue-creation` if unclear
- "Can you look at X" → `investigation` (curiosity/question) or `issue-creation` (symptom/bug)
- "Implement X" / "Add X" → feature-impl (direct or via daemon)

Beads issues are **per-repo**. `orch spawn --issue <id>` only works if the issue exists in the current repo's `.beads/` directory.

**Common failure pattern:**
```bash
# In orch-knowledge directory
bd ready  # Shows meta-orchestration-xyz
cd ~/orch-cli && orch spawn --issue meta-orchestration-xyz  # FAILS - issue not found
```

**Solution:** Spawn ad-hoc instead:
```bash
cd ~/orch-cli && orch spawn feature-impl "task description from beads issue"
```

Then manually close the beads issue after completion:
```bash
cd ~/orch-knowledge && bd close meta-orchestration-xyz --reason "Completed in orch-cli"
```

**Always document the decision:**
For significant choices, create decision document. For routine work, state reasoning in git commit message.

## Models as Understanding Artifacts (NEW - Jan 2026)

**Purpose:** Synthesize investigation clusters into queryable understanding

**Status:** Validated at N=11 (Jan 2026). Still establishing boundaries (scale limits, staleness patterns, evolution frequency). Track effectiveness over 30-day validation period.

**What models are:** Operational representations of system understanding that compress investigation findings into queryable constraints and behaviors. Enables strategic questions ("What does X enable/constrain?") to be answered in <60s vs hours of investigation reading.

**Why models matter:** Investigation clusters signal missing coherent understanding. 87 investigations on "spawn" means answering "why does spawn fail if X?" requires reading many files. A model externalizes that understanding once, making it queryable.

---

### When to Create Models

**Trigger:** Investigation cluster detected (15+ investigations on single topic)

**Four-factor test (all required):**
1. **HOT** - Cluster exists (15+ investigations on topic)
2. **COMPLEX** - Has failure modes, constraints, state transitions (not simple CRUD)
3. **OWNED** - Our system internals (orch-go, spawn, daemon) - not external tools (tmux, kb CLI)
4. **STRATEGIC_VALUE** - "Enable/constrain" answers save hours vs minutes

**Decision:**
- **If all four factors met:** Spawn architect to create model
- **If missing factors:** Synthesize into guide (reusable process) or decision (architectural choice) instead

**Example - passing all four:**
- Topic: "spawn" (87 investigations) ✅ HOT
- Has failure modes (no issue, wrong context), constraints (must have .beads/), state (creating → pending → running) ✅ COMPLEX
- orch-go owns spawn.py ✅ OWNED
- "What does spawn enable/constrain?" is strategic planning question ✅ STRATEGIC_VALUE
- **Action:** Create `spawn-architecture.md` model

**Example - failing OWNED:**
- Topic: "tmux" (20 investigations) ✅ HOT
- Has state, failure modes ✅ COMPLEX
- We don't own tmux internals ❌ OWNED
- **Action:** Create guide or kb quick constraints, not model

- "Why does X fail?" → Check model's Why This Fails section

- "What are integration points?" → Check model's Integration Points section

**Location:** `.kb/models/{topic}.md`

**Discovery:** Models surface via `kb context` (same as investigations/decisions)

### Model Template Structure

Models use consistent 6-section structure (validated across N=11):

1. **What This Is** - Purpose, scope, what problem this solves
2. **How This Works** - Mechanisms, what this enables (state diagrams, flow charts)
3. **Why This Fails** - Failure modes, what breaks and why
4. **Constraints** - What this constrains/disallows, boundaries
5. **Integration Points** - How this connects to other systems
6. **Evolution** - Timestamped history of what changed and why

**Enable/Constrain Query:** The template embeds this structural query:
- "How This Works" section shows: what became possible
- "Constraints" section shows: what's no longer allowed/possible

When creating models, explicitly enumerate both sides of this query.

## Pressure Over Compensation (Human-System Interaction)

**Principle:** When the system fails to surface knowledge, don't compensate by providing it manually. Let the failure create pressure to improve the system.

**The test:** "Am I helping the system, or preventing it from learning?"

**What this means:**
- Orchestrator doesn't know something it should → Don't paste the answer
- Let the failure surface → That failure is data
- Ask "Why didn't the system surface this?" → Build the mechanism

**What this rejects:**
- "Here, let me paste the context you need" (compensates for broken surfacing)
- "I'll just tell you what you should already know" (human becomes the memory)
- "The system doesn't know, so I'll fill in" (prevents system improvement)

**The pattern:**

```
Human compensates for gap → System never learns → Human keeps compensating
Human lets system fail → Failure surfaces gap → Gap becomes issue → System improves
```

**The workflow when context is missing:**

1. **Don't paste** - Resist the urge to provide missing context manually
2. **Note the gap** - What knowledge should the system have surfaced?
3. **Let the failure be felt** - Agent struggles or produces suboptimal work
4. **Create improvement** - Use `orch learn` to track gaps or create beads issue
5. **Build the mechanism** - OpenCode plugin (session.created event), skill update, kb quick entry, etc.

**When to apply this:**
- Orchestrator asks about something that should be in kb
- Agent spawns missing context that prior agents needed
- Same context gap appears in multiple sessions
- Human finds themselves typing the same explanations repeatedly

**Exceptions (when compensating is acceptable):**
- Truly novel context that couldn't have been predicted
- One-off task-specific details (not reusable)
- Emergency situations where waiting isn't viable

**Integration with orch learn:**
Context gaps detected during spawns are tracked automatically. Run `orch learn` to see recurring gaps and act on them:

```bash
orch learn                    # See suggestions
orch learn act 1              # Run first suggestion
orch learn effects            # Check if improvements helped
```

**Why this is foundational:** Session Amnesia says agents forget. This principle says *don't be the memory*. Your role is to create pressure that forces the system to develop its own memory mechanisms. Every time you compensate, you relieve the pressure and the system stays broken.

**Reference:** `.kb/decisions/2025-12-25-pressure-over-compensation.md`, `~/.kb/principles.md`

### Proactive Hygiene Checkpoint

**Purpose:** Systematic backlog cleanup prevents duplicate spawns, stale issue accumulation, and priority drift.

**When to run hygiene checkpoint:**

| Trigger | When | Why |
|---------|------|-----|
| **Session start** | Before diving into work | Orient on priorities AND clean backlog |
| **Before batch-labeling** | Before marking 3+ issues `triage:ready` | Avoid spawning duplicates |
| **Weekly rhythm** | Every 7 days | Prevent drift and accumulation |
| **Friction signal** | Backlog feels noisy (>30 open, >10 P1) | Reactive cleanup when needed |

**Hygiene workflow:**

1. **Dedup:** `bd duplicates` - Find exact duplicates
   - Review suggested merges
   - Run `bd duplicates --auto-merge` if confident, or merge manually via `bd duplicate <dupe-id> <target-id>`

2. **Stale:** `bd stale --days 30` - Find abandoned/forgotten issues
   - Close if no longer relevant: `bd close <id> --reason "stale"`
   - Defer if still relevant but not now: `bd defer <id>`
   - Update if still active but forgot to touch: `bd comment <id> "still active"`

3. **Epic cleanup:** `bd epic status` - Check epic progress
   - Identify orphan issues that should be epic children
   - Add to epics: `bd dep add <epic-id> <child-id>`
   - Close completed epics: `bd epic close-eligible`

4. **Reprioritize:** `bd list --priority P1,P2` - Review high-priority queue
   - Adjust priorities based on new context
   - Unblock blocked issues if dependencies resolved

**Quick hygiene (session start, 2-3 minutes):**
```bash
bd duplicates              # Any exact dupes?
bd stale --days 30 --limit 5   # Top 5 stale issues?
bd epic status             # Any epics ready to close?
```

**Full hygiene (weekly, 10-15 minutes):**
```bash
bd duplicates --auto-merge     # Auto-merge exact matches
bd stale --days 30             # Review all stale
bd epic status                 # Epic health check
bd list --priority P1 --status open   # P1 queue audit
```

**Why this matters:**
- Prevents duplicate spawn work (wasted agent time)
- Surfaces forgotten priorities (what fell through cracks?)
- Keeps backlog signal-rich (noise drowns out signal)
- Reduces orchestrator cognitive load (fewer distractions)

**Anti-patterns:**
- ❌ Skipping hygiene because "just this once" (technical debt compounds)
- ❌ Batch-labeling without dedup check (high risk of duplicate spawns)
- ❌ Letting stale issues accumulate past 50+ (backlog becomes unusable)
- ❌ Closing stale issues without review (may lose important context)

## Skill Selection Guide

### Strategic-First Orchestration

**Principle:** In hotspot areas (persistent failure patterns), architect is the default. Tactical debugging requires `--force` with justification.

**Hotspot detection:** Areas with 5+ bug fixes, 3+ investigations, or 2+ abandonments in the same component signal systemic issues, not isolated bugs.

**The gate:**
- When spawning to a hotspot area → `architect` skill required (understand root cause)
- To use tactical skills (systematic-debugging, feature-impl) → Must use `--force` flag + provide justification
- Prevents retry loops: tactical fixes to systemic problems waste agent time

**Why this matters:**
- Hotspots reveal systemic issues that tactical debugging won't solve
- 5th bug fix to same area = premise is wrong, not implementation
- Strategic approach (architect) identifies root cause before spawning fixes

**Example workflow:**
```bash
# Hotspot detected in auth middleware (6 fixes, 4 investigations)
orch spawn architect "understand auth middleware failure pattern"
# After architect identifies root cause:
orch spawn feature-impl "refactor auth based on architect findings"

# Override (requires justification):
orch spawn --force systematic-debugging "auth bug" 
# Must document why tactical approach is appropriate despite hotspot
```

**Reference:** `.kb/decisions/2026-01-11-strategic-first-orchestration.md`

## Progressive Handoff Documentation

**Principle:** Capture at Context - forcing functions must fire when context exists, not just before completion. Context decays.

**The anti-pattern:** Wait until session end, try to recall everything, produce incomplete handoff. This captures *reconstructed* context, not *observed* context.

**The discipline:** Fill in SESSION_HANDOFF.md progressively AS YOU WORK, not at the end.

**Location:** `.orch/session/{window}/active/SESSION_HANDOFF.md` (created by `orch session start`)

### Update Triggers

| Trigger | What to Update | Example |
|---------|----------------|---------|
| **After session start (first 5 tool calls)** | TLDR, "Where We Started" | Initial framing, current state |
| **After each spawn completes** | Spawns table, Evidence | Add row with outcome, key finding |
| **When friction occurs** | Friction section | "orch session end didn't prompt" |
| **When making a decision** | Knowledge section | "Chose X because Y" |
| **Before `orch session end`** | Next section, verify complete | Recommendations for next session |

### Pre-Session-End Gate

**Before running `orch session end`, verify:**
- [ ] TLDR reflects what actually happened (not initial framing)
- [ ] Spawns table has all agents with outcomes
- [ ] Knowledge section captures decisions and constraints discovered
- [ ] Friction section documents what was harder than expected
- [ ] Next section has actionable recommendations

**If handoff is incomplete:** Update it FIRST, then run `orch session end`.

### Why This Matters

`orch session end` may not prompt for input (stdin unavailable in spawned contexts). Even when it does prompt, recall at end is unreliable. Progressive documentation:
- Captures details while fresh
- Survives session crashes
- Enables resume even if session end fails
- Creates better handoffs for next orchestrator

## Session Resume Protocol

**Purpose:** Automatic session continuity through handoff injection - orchestrators don't need to remember session mechanics.

**Quick reference:**
```bash
# Manual resume (review handoff before starting work)
orch session resume

# Check if handoff exists (scripting)
orch session resume --check

# End session (creates handoff for next session)
orch session end
```

**Automatic behavior:** When starting a new Claude Code or OpenCode session, hooks automatically inject the latest handoff if one exists. Zero cognitive load - you don't need to remember to resume.

**How it works:**
1. Session start hook runs: `~/.claude/hooks/session-start.sh` or `~/.config/opencode/plugin/session-resume.js`
2. Discovers handoff by walking up tree to find `{project}/.orch/session/latest/SESSION_HANDOFF.md`
3. Injects content into fresh Claude instance (or silent if no handoff - fresh starts are valid)

**File structure (project-specific):**
```
{project}/.orch/session/
├── latest -> 2026-01-13-0827/    # Symlink to most recent
├── 2026-01-11-1935/
│   └── SESSION_HANDOFF.md
└── 2026-01-13-0827/
    └── SESSION_HANDOFF.md
```

**Multi-project support:** Each repo maintains its own session history. Working in orch-go discovers orch-go handoffs only, even if you have active sessions in other projects.

**Creating handoffs:**
- **Automatic:** `orch session end` (part of session close protocol - step 6 of 7)
- Creates timestamped directory with SESSION_HANDOFF.md
- Updates `latest` symlink automatically
- Prompts for reflection (goal, accomplishments, friction, next priorities)

**Common workflows:**

| Scenario | What Happens | Valid? |
|----------|--------------|--------|
| Fresh start (no handoff) | Hook finds nothing, stays silent | ✅ Yes - fresh starts expected |
| Resume after break | Hook auto-injects latest handoff | ✅ Yes - zero cognitive load |
| Manual review | Run `orch session resume` before starting session | ✅ Yes - verify relevance |

**Troubleshooting:**
- **"No handoff found"** → Check `.orch/session/latest` symlink exists and points correctly
- **Hook not injecting** → Verify hook file exists, is executable, and `orch` command in PATH
- **Hook runs but handoff doesn't appear (Claude Code)** → Output must be JSON format (see `~/.claude/hooks/session-start.sh` lines 10-22), AND hook must be registered in `~/.claude/settings.json` (not just `cdd-hooks.json`)

**Session close protocol integration:**
```
[ ] 1. git status
[ ] 2. git add <files>
[ ] 3. bd sync
[ ] 4. git commit -m "..."
[ ] 5. bd sync
[ ] 6. orch session end        ← CREATES HANDOFF (enables resume)
[ ] 7. git push
```

**Key insight:** Session resume is a forcing function. Running `orch session end` (step 6) creates the handoff that enables automatic resume in the next session. Skipping this step breaks continuity.

**Full reference:** `.kb/guides/session-resume-protocol.md` - Complete documentation with edge cases, discovery logic, and implementation details
