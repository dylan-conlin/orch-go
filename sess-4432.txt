‚Źļ No, overmind is NOT buggy - it's a solid tool. The issue is launchd's environment handling, not overmind itself.

  Let me explain what's actually happening:

  The Technical Issue

  When you run overmind manually:
  $ overmind start -D
  # Works fine - inherits your shell's PATH

  When launchd runs overmind:
  <key>EnvironmentVariables</key>
  <dict>
      <key>PATH</key>
      <string>/Users/dylanconlin/.bun/bin:/opt/homebrew/bin:...</string>
  </dict>
  - launchd sets PATH as requested
  - But when overmind spawns tmux, the PATH gets lost
  - This is because overmind uses tmux's -C control mode which has its own environment handling

  Why This Happens

  Overmind creates tmux sessions with commands like:
  tmux -C -L overmind-orch-go-xyz new -n api ...

  The tmux control mode (-C) starts with a minimal environment. The PATH from launchd's EnvironmentVariables doesn't propagate through to tmux's subprocess spawning.

  This is a known interaction pattern between launchd ‚Üí overmind ‚Üí tmux, not a bug in any single tool.

  The .env File Attempt

  I created a .env file with PATH thinking overmind would read it (some tools do). But that didn't work because:
  - Overmind reads .env and passes it to the processes it manages (api, web, opencode)
  - But overmind needs tmux BEFORE it can manage processes
  - So the PATH in .env doesn't help overmind find tmux

  Why I'm Not Recommending Overmind for This

  It's not that overmind is bad. It's that:

  1. The problem is solvable but fiddly - we'd need to:
    - Create a wrapper script that sets PATH
    - Or use a different process manager (foreman) that launchd handles better
    - Or abandon launchd and use tmuxinator directly
  2. We don't actually need launchd supervision for overmind because:
    - Overmind itself is stable (doesn't crash)
    - The services crash sometimes, but overmind auto-restarts them
    - If overmind dies, we just restart it manually (rare)
  3. We have a better alternative - orch doctor --daemon:
    - Monitors services directly (ports 4096, 3348, 5188)
    - Kills orphans (long-running bd processes)
    - Restarts crashed services
    - Doesn't need tmux, runs as simple Go process
    - Can be supervised by launchd easily (no tmux dependency)

  What I'm Actually Recommending

  Keep using overmind - it works great when started manually:
  overmind start -D  # In your shell startup or manually

  Add launchd supervision for orch doctor --daemon instead:
  <!-- ~/Library/LaunchAgents/com.orch.doctor.plist -->

      <string>/Users/dylanconlin/bin/orch</string>

  </array>

  This gives us:
  - Overmind for service management (start/stop/restart)
  - orch doctor --daemon for monitoring and self-healing
  - No launchd ‚Üí overmind ‚Üí tmux PATH issues

  Does the epic need "overmind supervised by launchd" or does it need "services stay healthy automatically"?

  The Gap

  Doctor daemon can detect but can't restart because:
  - Detection works (polls ports every 30s) ‚úÖ
  - Restart logic assumes launchd services exist ‚ĚĆ
  - Our services are managed by overmind, not launchd
  - When overmind isn't running (tmux PATH issue), nothing can restart services

  What This Means

  Current architecture provides:
  - ‚úÖ Orphan cleanup (long-running bd processes)
  - ‚úÖ Crash detection (within 30s)
  - ‚ĚĆ Auto-restart (requires overmind or launchd services)

  The choice:

  Option A: Accept this limitation
  - Manual restart via orch deploy when crashes detected
  - Doctor daemon alerts you via desktop notification
  - Overmind usually stable, crashes rare

  Option B: Fix restart capability
  - Add restart logic to doctor daemon: overmind restart <service>
  - Requires overmind to be running
  - But overmind has tmux PATH issue...

  This is circular: we need overmind for restarts, but overmind needs launchd supervision for reliability, but launchd can't run overmind due to tmux PATH.
